  ---
  关键发现：rv_resource_cards 根本不会自己滚动

  看 fragment_my_page.xml:84-88：

  <androidx.recyclerview.widget.RecyclerView
      android:id="@+id/rv_resource_cards"
      android:layout_width="match_parent"
      android:layout_height="wrap_content"
      android:nestedScrollingEnabled="false" />   ← 两个关键属性

  两个属性共同决定了命运：

  ┌────────────────────────────────┬─────────────────────────────────────────┐
  │              属性              │                  效果                   │
  ├────────────────────────────────┼─────────────────────────────────────────┤
  │ wrap_content                   │ RV 一次性展开所有 item，自身内容高度 =  │
  │                                │ 全部条目，不需要内部滚动                │
  ├────────────────────────────────┼─────────────────────────────────────────┤
  │ nestedScrollingEnabled="false" │ RV 彻底退出嵌套滑动协议                 │
  └────────────────────────────────┴─────────────────────────────────────────┘

  结论：rv_resource_cards 永远不会自己滚动。 它是被 NestedScrollableContainer
  通过 scrollBy()/scrollTo() 整体移动的，RV 内部的 scrollY 始终是 0。

  ---
  回答你的三个问题

  Q1：addOnScrollListener 还能正常工作吗？

  不能。 RV 自身不滚动，onScrollStateChanged 和 onScrolled 永远不会触发。

  Q2：需要用 NestedScroll 的回调吗？

  不需要，因为 nestedScrollingEnabled=false 已经关掉了。正确的监听点是
  NestedScrollableContainer 本身的 scrollY 变化。

  ---
  具体实现方案

  最简洁可靠的方法：在 NestedScrollableContainer 里重写 onScrollChanged，加
  Handler 防抖：

  NestedScrollableContainer.kt 里添加：

  // 在类顶部加这几行
  private val scrollStopHandler =
  android.os.Handler(android.os.Looper.getMainLooper())
  private val scrollStopRunnable = Runnable { onScrollStopped?.invoke() }
  var onScrollStopped: (() -> Unit)? = null

  // 重写 onScrollChanged（LinearLayout 已有此回调，直接 override）
  override fun onScrollChanged(l: Int, t: Int, oldl: Int, oldt: Int) {
      super.onScrollChanged(l, t, oldl, oldt)
      scrollStopHandler.removeCallbacks(scrollStopRunnable)
      scrollStopHandler.postDelayed(scrollStopRunnable, 150)
  }

  调用方（Fragment/Activity）：

  nestedContainer.onScrollStopped = {
      // 滑动停止后的回调，此时 rv_resource_cards 已静止
      // 可做：曝光打点、图片懒加载触发、刷新某个状态等
  }

  为什么这个方案可以覆盖所有场景

  │ fling 转移到 feed      │                                │ ❌ 不再触发 → │
  │ RV（abortFling）       │ ❌ 停止变化                    │  防抖到期 →   │
