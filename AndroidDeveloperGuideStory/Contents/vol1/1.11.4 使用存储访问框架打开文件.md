### SAF是什么？

黛琳翻开书，指着上面的图示："SAF是Android 4.4（API 19）引入的框架。它的核心思想是——让用户通过一个统一的界面，能访问所有地方的文件。"

"所有地方？"洛芙问。

"对，本地的、云端的、U盘的……只要实现了SAF接口的存储服务，都可以通过这个统一的界面访问。"黛琳画出结构图：

```
┌─────────────────────────────────────────────┐
│              Storage Access Framework        │
├─────────────────────────────────────────────┤
│                                             │
│  ┌─────────────┐   ┌─────────────┐        │
│  │  Picker     │   │  Client App │        │
│  │  (系统界面) │   │  (我们的)    │        │
│  └──────┬──────┘   └──────┬──────┘        │
│         │                  │               │
│         │    Intent        │               │
│         │◄─────────────────┤               │
│         │                  │               │
│  ┌──────▼──────┐   ┌──────▼──────┐        │
│  │  Document   │   │DocumentsProvider│      │

│  └─────────────┘   └─────────────┘        │
└─────────────────────────────────────────────┘
```

"这个Picker，就是那个系统统一的文件选择界面，"希尔补充，"用户打开后，可以看到所有接入了SAF的云服务——Google Drive、OneDrive、本地存储……想从哪选就从哪选。"

### 三大核心组件

伊莎不知什么时候也凑了过来，指着图说："我来理一理——这个框架里有三个东西："

"**Document Provider（文档提供者）**，"伊莎扳着手指，"就是那个实现了DocumentsProvider的东西。云盘啊、本地文件啊，都属于这一类。"

"**Client App（客户端应用）**，"希尔接话，"就是我们的app，通过发Intent来让用户选文件。"

"**Picker（选择器）**，"黛琳总结，"就是系统弹出来的那个统一界面。用户在里面选，完了把结果返回给我们。"

### 第一个例子：打开文件

希尔把屏幕转过来："看，最简单的用法——让用户选一个文件："

```kotlin
// 让用户选择一个文件（Android 4.4+）
// 这个Intent会打开系统Picker
val intent = Intent(Intent.ACTION_OPEN_DOCUMENT).apply {
    // 筛选：只要图片类型的文件
    addCategory(Intent.CATEGORY_OPENABLE)
    type = "image/*"
}
startActivityForResult(intent, REQUEST_CODE_OPEN_IMAGE)
```

"然后在onActivityResult里处理结果："希尔继续写：

```kotlin
override fun onActivityResult(requestCode: Int, resultCode: Int, data: Intent?) {
    super.onActivityResult(requestCode, resultCode, data)

    if (requestCode == REQUEST_CODE_OPEN_IMAGE && resultCode == RESULT_OK) {
        // 用户选中的文件URI
        val uri = data?.data
        uri?.let {
            // 有永久读取这个文件的权限
            // 官方推荐：获取持久化权限
            val takeFlags = Intent.FLAG_GRANT_READ_URI_PERMISSION
            contentResolver.takePersistableUriPermission(it, takeFlags)

            // 接下来就可以读取这个文件了
            val inputStream = contentResolver.openInputStream(it)
            // 处理文件内容...
        }
    }
}
```

"用户选了图片以后，"希尔解释，"我们不仅能读取这个图片，还能'记住'这个选择——这就是官方说的'持久化权限'。"

### 持久化权限：像书签一样

"持久化权限是什么？"洛芙问。

"你想啊，"伊莎比划着，"用户今天选了一张照片，明天再打开app，如果还要再选一次，是不是很烦？"

洛芙点头："对哦。"

"持久化权限就是解决这个问题，"黛琳说，"我们把用户选的URI'记下来'，下次直接用："

```kotlin
// 读取之前保存的URI
val prefs = getSharedPreferences("my_prefs", Context.MODE_PRIVATE)
val savedUri = prefs.getString("saved_image_uri", null)

if (savedUri != null) {
    val uri = Uri.parse(savedUri)
    try {
        // 重新获取权限（因为权限可能会过期）
        val takeFlags = Intent.FLAG_GRANT_READ_URI_PERMISSION
        contentResolver.takePersistableUriPermission(uri, takeFlags)

        // 打开文件
        val inputStream = contentResolver.openInputStream(uri)
        // ...
    } catch (e: SecurityException) {
        // 权限过期了，让用户重新选
        openFile()
    }
}
```

### 创建新文件：ACTION_CREATE_DOCUMENT

"如果不仅是读取，还要创建新文件呢？"洛芙问。

"用ACTION_CREATE_DOCUMENT，"希尔又写：

```kotlin
// 让用户创建一个新文件
val intent = Intent(Intent.ACTION_CREATE_DOCUMENT).apply {
    addCategory(Intent.CATEGORY_OPENABLE)
    // 指定MIME类型（这里是文本文件）

startActivityForResult(intent, REQUEST_CODE_CREATE_FILE)
```

"用户会看到一个'保存为'的界面，"希尔解释，"可以选地方、输入文件名，然后创建文件。"

### 选择整个目录：ACTION_OPEN_DOCUMENT_TREE

"有时候我们不是要选一个文件，而是要选一整个文件夹，"黛琳说，"比如备份app想让我们选一个备份目录。"

"那个更厉害——ACTION_OPEN_DOCUMENT_TREE，"希尔说，"Android 5.0（API 21）以上才支持："

```kotlin
// 让用户选择一个目录
val intent = Intent(Intent.ACTION_OPEN_DOCUMENT_TREE)
startActivityForResult(intent, REQUEST_CODE_PICK_DIRECTORY)
```

"选完目录后，"希尔继续，"我们可以在这个目录里任意创建、读取、删除文件："

```kotlin
override fun onActivityResult(requestCode: Int, resultCode: Int, data: Intent?) {
    if (requestCode == REQUEST_CODE_PICK_DIRECTORY && resultCode == RESULT_OK) {
        val treeUri = data?.data

        treeUri?.let { uri ->
            // 获取对这个目录的持久化权限
            val takeFlags = Intent.FLAG_GRANT_READ_URI_PERMISSION or
                          Intent.FLAG_GRANT_WRITE_URI_PERMISSION
            contentResolver.takePersistableUriPermission(uri, takeFlags)

            // 可以在这个目录下创建文件
            val newFileUri = DocumentsContract.createDocument(
                contentResolver,
                treeUri,
                "text/plain",
                "my_file.txt"
            )
        }
    }
}
```

### Document Provider的数据模型

"说了半天client端，"黛琳翻到新的一页，"我们来讲讲Provider端的数据模型。这是SAF的核心——"

```kotlin
// Document Provider 的数据模型
// 基于传统的文件层次结构

// 1. 每个Provider有一个或多个Root（根）
//    - 每个Root有一个唯一的COLUMN_ROOT_ID
//    - Root可以理解为"驱动器"：本地存储、Google Drive、USB等
//    - Root是动态的：USB插上就出现，拔掉就消失

// 2. 每个Root指向一个Document（目录）
//    - 这个Document就是进入这个Root的入口

// 3. 每个Document可以指向N个其他Document
//    - 可以是文件（有具体的MIME类型）
//    - 也可以是目录（MIME类型是MIME_TYPE_DIR）

// 4. 每个Document有一个唯一的COLUMN_DOCUMENT_ID
//    - 这个ID是永久的，不会变
//    - 即使设备重启，ID也保持有效

// 5. 每个Document可以有不同的能力（FLAGS）
//    - FLAG_SUPPORTS_WRITE：可写
//    - FLAG_SUPPORTS_DELETE：可删除
//    - FLAG_SUPPORTS_THUMBNAIL：支持缩略图
```

洛芙听得入神："这不就是文件系统吗？"

"对，"黛琳微笑，"SAF就是用Content Provider的方式，把文件系统抽象了一层。不管你底层是本地存储、云存储、甚至是一个全新的存储方式，只要实现DocumentsProvider，就能接入这个统一的界面。"

### 官方示例：看图应用

"我们来看一个实际例子，"希尔打开一个假想的照片应用：

```kotlin
// 一个照片应用的SAF使用流程

class PhotoAppActivity : AppCompatActivity() {

    // 1. 打开图片
    fun openImage() {
        val intent = Intent(Intent.ACTION_OPEN_DOCUMENT).apply {
            addCategory(Intent.CATEGORY_OPENABLE)
            type = "image/*"  // 只要图片
        }
        startActivityForResult(intent, REQUEST_OPEN_IMAGE)
    }

    // 2. 处理结果
    override fun onActivityResult(request: Int, result: Int, data: Intent?) {
        super.onActivityResult(request, result, data)

        if (result == RESULT_OK) {
            val imageUri = data?.data
            imageUri?.let { uri ->
                // 获取持久化权限
                val flags = Intent.FLAG_GRANT_READ_URI_PERMISSION
                contentResolver.takePersistableUriPermission(uri, flags)

                // 显示图片
                showImage(uri)
            }
        }
    }

    // 3. 保存URI供下次使用
    fun saveImageUri(uri: Uri) {
        getSharedPreferences("photos", MODE_PRIVATE)
            .edit()
            .putString("last_image", uri.toString())
            .apply()
    }

    // 4. 恢复图片
    fun restoreImage(): Uri? {
        val uriStr = getSharedPreferences("photos", MODE_PRIVATE)
            .getString("last_image", null)
        return uriStr?.let { Uri.parse(it) }
    }
}
```

"这个流程很清晰吧？"希尔说，"用户选图片 -> 我们拿到URI -> 获取权限 -> 显示 -> 保存URI下次用。"

### 技术总结

> **使用存储访问框架打开文件** —— 本章详细讲解了Android的Storage Access Framework (SAF)，包括三大核心组件（Document Provider、Client App、Picker）、三种Intent用法（ACTION_OPEN_DOCUMENT、ACTION_CREATE_DOCUMENT、ACTION_OPEN_DOCUMENT_TREE）、持久化权限的实现，以及Document Provider的数据模型。SAF让应用能够以统一的方式访问任意存储服务的文件。

#### 今日关键词

1. **SAF (Storage Access Framework)**：存储访问框架，Android 4.4引入
2. **ACTION_OPEN_DOCUMENT**：打开文件（4.4+）
3. **ACTION_CREATE_DOCUMENT**：创建文件（4.4+）
4. **ACTION_OPEN_DOCUMENT_TREE**：选择目录（5.0+）
5. **takePersistableUriPermission**：获取持久化权限
6. **DocumentsProvider**：文档提供者的基类

#### SAF的三大组件

| 组件 | 作用 |
|------|------|
| Document Provider | 存储服务的实现（如Google Drive） |
| Client App | 发起请求的应用 |
| Picker | 系统统一的文件选择界面 |

#### Intent对比

| Intent | 作用 | 版本 |
|--------|------|------|
| ACTION_PICK / ACTION_GET_CONTENT | 选择单个文件 | 全部 |
| ACTION_OPEN_DOCUMENT | 选择文件（带系统Picker） | 4.4+ |
| ACTION_CREATE_DOCUMENT | 创建新文件 | 4.4+ |
| ACTION_OPEN_DOCUMENT_TREE | 选择目录 | 5.0+ |

#### 设计思想

- **统一抽象**：不管底层是云还是本地，对上接口一致
- **用户中心**：统一的Picker，用户体验一致
- **持久化**：一次授权，长期有效
- **安全**：URI权限需要显式授予

#### 反模式与陷阱

* **要求获取 WRITE_EXTERNAL_STORAGE 来读取用户选择的云端图片**：
  * **修复方案**：直接使用 `ACTION_OPEN_DOCUMENT`，用户的点击就能授予所选文件的临时/持久权限，彻底绕过危险的宽泛权限。
* **忘记处理权限失效**：
  * **修复方案**：即使有了持久化权限，底层的云端文件也可能被外部删除。始终用 `try-catch` 包裹 `openInputStream` 以捕获 `SecurityException` 或 `FileNotFoundException`。
* **使用 File(uri.path) 硬解析提取文件路径**：
  * **修复方案**：SAF 返回的 Uri 是 `content://` 协议的，不允许直接当作传统的 `java.io.File` 解析，必须使用 `contentResolver.openInputStream(uri)`。

#### 参考实现要点

* **权限生命周期管理**：SAF 赋予的长效权限和当前系统记录周期挂钩，如果未明确固化 `takePersistableUriPermission`，所在当前 Activity 结束进程被挥发后访问权将会剥夺。
* **持久化读写修饰符限制**：固化权限时传入的标志 `FLAG_GRANT_READ_URI_PERMISSION`，必须和启动该 Picker 返回给予的最原始权利相符或缩减，切莫扩充请求。
* **不要信赖扩展名**：要信赖 `ContentResolver.getType(uri)` 和内部查询的元数据去判断，永远不要试图正则匹配 Uri 中最后携带的字符扩展名来当作唯一类型。
