## 1.4.1 管理存储设备上的所有文件

烈日当空，蝉鸣声如潮水般涌来。在那片被称作“回音谷”的山坳里，露营旅团正在进行午休。

洛芙坐在一块干燥的岩石上，手里拿着平板电脑，眉头紧锁。“真是的……”她嘟囔着，“我写在这个‘垃圾清理大师’Demo 怎么完全不听话？明明检测到了下载文件夹里有一堆乱七八糟的临时文件（`.tmp`），可每次尝试删除，系统就报 `Permission Denied`。”

“因为那是别人留下的垃圾呀，”伊莎正用湿毛巾擦拭着额头的汗水，“在 Android 的世界里，每个应用都守着自己的领地。现在的分区存储（Scoped Storage）规则很严格，你只能清理自己的房间，不能跑到公共走廊去把别人的鞋子扔掉——除非那是你自己生成的。”

“可是做‘清理软件’不就是为了打扫公共区域吗？”洛芙不甘心地戳着屏幕，“如果没有权限，我的 App 还有什么意义？”

黛琳放下了手中的地图，推了推眼镜，神色变得严肃起来：“确实，对于普通应用来说，随意访问所有文件是不被允许的。但对于像文件管理器、备份工具或者杀毒软件这类的特殊应用，Android 确实留了一把——**万能钥匙**。”

“万能钥匙？！”洛芙的耳朵竖了起来。

“它的名字叫 `MANAGE_EXTERNAL_STORAGE`，”希尔把最后一口三明治咽下去，打了个响指，“也就是‘所有文件访问权限’。一旦拥有了它，你就能像以前一样，用标准的 `java.io.File` API 访问绝大多数文件，不再受 MediaStore 的限制。”

洛芙兴奋地从岩石上跳下来：“那还等什么？快教我怎么拿这把钥匙！”

“别急，”黛琳按住了她的肩膀，“这把钥匙太沉重了。Google Play 对它有极严格的审核政策。如果你的应用核心功能不需要它，只是想图省事，那审核是绝对过不了的。你确定要承担这份责任吗？”

“我要！”洛芙坚定地点点头，“我都写了‘清理大师’了，当然要清理得干干净净！”

### 第一步：声明你的野心 (Manifest)

“首先，你得在清单文件里大声喊出来，”希尔指着代码说，“告诉系统你需要这个超级权限。”

```xml
<!-- AndroidManifest.xml -->
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.campcleaner">

    <!-- 这是一个特殊权限，不属于运行时权限组，而是一个“AppOps”操作 -->
    <uses-permission android:name="android.permission.MANAGE_EXTERNAL_STORAGE" />

</manifest>
```

“注意，”希尔补充道，“这个权限不仅需要在 Manifest 里写，还需要用户手动去系统设置的一个深层页面里授权。它不像那种弹个窗就能点的普通权限。”

### 第二步：检查与申请 (Check and Request)

“因为这个权限太危险，系统不会自动弹窗询问。”黛琳解释道，“你需要自己引导用户去设置页面。”

```kotlin
// 1. 检查是否已经拥有"所有文件访问权限"
// Environment.isExternalStorageManager() 是 Android 11+ 引入的 API
fun checkPermission(): Boolean {
    return if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.R) {
        Environment.isExternalStorageManager()
    } else {
        // Android 10 及以下通常使用 READ/WRITE_EXTERNAL_STORAGE
        // 这里为了简化，我们只讨论 Android 11+ 的核心场景
        true 
    }
}

// 2. 引导用户去授权页面
fun requestPermission(context: Context) {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.R) {
        try {
            //以此 Intent 打开系统设置页面，直达本应用的授权页
            val intent = Intent(Settings.ACTION_MANAGE_APP_ALL_FILES_ACCESS_PERMISSION)
            intent.addCategory("android.intent.category.DEFAULT")
            // 传递包名，让设置页面直接定位到我们 App
            intent.data = Uri.parse("package:${context.packageName}")
            context.startActivity(intent)
        } catch (e: Exception) {
            // 万一有的ROM太奇葩，不支持上面的 Intent，就跳转到总列表
            val intent = Intent(Settings.ACTION_MANAGE_ALL_FILES_ACCESS_PERMISSION)
            context.startActivity(intent)
        }
    } else {
        // 旧版本逻辑...
    }
}
```

洛芙看着那个复杂的 Intent Action，撇了撇嘴：“藏得可真深啊。”

“这是为了防止用户被诱导误点，”伊莎笑着说，“就像保管一把威力巨大的魔法杖，取用的时候总得经过三重封印嘛。”

### 第三步：自由的文件操作 (File Operations)

“还是原来的配方，还是原来的味道。”希尔一边敲代码一边说，“一旦拿到了这个权限，你就可以抛弃 MediaStore，直接用最原始、最简单的 `File` 对象来操作了。还记得 Java 课上学的那些吗？”

```kotlin
// 拥有权限后，列出 Download 文件夹下的所有文件（包括其他应用创建的！）
fun cleanUpDownloads() {
    val downloadsDir = Environment.getExternalStoragePublicDirectory(Environment.DIRECTORY_DOWNLOADS)
    
    // 直接使用 File API
    val files = downloadsDir.listFiles()
    
    if (files != null) {
        for (file in files) {
            // 比如，我们要删除所有的 .tmp 临时文件
            if (file.isFile && file.name.endsWith(".tmp")) {
                val success = file.delete() // 简单粗暴，直接删除！
                if (success) {
                    Log.d("CampCleaner", "已清理垃圾文件: ${file.name}")
                } else {
                    Log.e("CampCleaner", "删除失败: ${file.name}")
                }
            }
        }
    }
}
```

“哇！”洛芙惊呼，“没有 `ContentResolver`，没有 `Uri`，没有 `Selection` 语句！直接 `file.delete()`！这也太爽了吧！”

“这就是‘万能钥匙’的魅力，”黛琳微笑着说，“简单，直接，但也仅仅属于少数被选中的应用。”

### 禁区中的禁区

“不过，”希尔话锋一转，表情变得严肃，“即使有了这把钥匙，也有些地反而你是绝对进不去的。”

“哪怕我是‘管理者’也不行？”

“不行。`Android/data/` 和 `Android/obb/` 这两个目录，在 Android 11+ 上即使是拥有 `MANAGE_EXTERNAL_STORAGE` 权限的应用，通常也无法直接访问（除了文件管理器等极少数豁免情况，且限制越来越严）。这是为了保护各个应用最私密的数据不被篡改。”

洛芙若有所思地点点头：“就像虽然我是营地管理员，可以清理公共广场的垃圾，但我绝不能闯进别人的帐篷里去翻东西。”

“这比喻非常精准。”伊莎赞许地点了点头。

午后的风变得热烈起来，吹得树叶哗哗作响。洛芙合上电脑，长长地伸了个懒腰，感觉自己像是一个剛剛拿到尚方宝剑的侠客，虽然手中沉甸甸的，但心里充满了守护营地清洁的使命感。拥有力量的感觉真好，但更重要的是，知道这力量该用在哪里。

### 技术总结

> **MANAGE_EXTERNAL_STORAGE**（所有文件访问权限）是 Android 11 (API 30) 引入的高级权限，授予应用对外部存储中绝大多数文件的**直接读写访问权**（就像旧版本的存储权限一样）。它允许使用原始的 `java.io.File` API，绕过 MediaStore 的限制。由于隐私风险极高，Google Play 对此权限有严格的上架审核政策（仅限文件管理、备份恢复、杀毒等核心功能应用）。

#### 今日关键词
1.  **MANAGE_EXTERNAL_STORAGE**：超级权限，允许读写公用目录下的所有文件。
2.  **Environment.isExternalStorageManager()**：检查是否已获得该权限的静态方法。
3.  **Settings.ACTION_MANAGE_APP_ALL_FILES_ACCESS_PERMISSION**：引导用户授权的 Intent Action。
4.  **Google Play Policy**：使用此权限的应用必须在 Play Console 声明核心用途，否则会被下架。

#### 结构图

```mermaid
flowchart TD
    Start[App 需要访问所有文件] --> Check{isExternalStorageManager()?}
    Check -->|Yes| Access[使用 File API 直接读写]
    Check -->|No| Request[跳转设置页面 Request Permission]
    
    Request --> UserAction[用户手动开关授权]
    UserAction -->|允许| Access
    UserAction -->|拒绝| Limited[只能访问自己的文件或 MediaStore]
    
    Access --> Scope((权限范围))
    Scope --> Public[Public Dir: Download, Documents...]
    Scope --> NoAccess[禁止访问: Android/data, Android/obb]
```

> 图 1：所有文件访问权限的申请与生效范围。

#### 反模式与陷阱

1.  **滥用权限**：普通图片编辑器申请此权限。
    *   **后果**：Google Play 审核被拒，用户不信任。
    *   **修复**：使用 MediaStore 或 SAF (系统文件选择器)。
2.  **忘记检查版本**：在 Android 10 以下直接跳转此 Settings 页面。
    *   **后果**：崩溃 (`ActivityNotFoundException`)。
    *   **修复**：检查 `Build.VERSION.SDK_INT >= Build.VERSION_CODES.R`。
3.  **试图访问 Android/data**：以为有了权限就能为所欲为。
    *   **后果**：`listFiles()` 返回 null 或 `Permission Denied`。
    *   **修复**：这是系统级限制，除非你是特定的文件管理器且通过特殊适配，否则不要尝试。

**目标**：编写一个工具类，安全地检查和请求权限。
**你需要做的事**：
1. 创建 `PermissionHelper` 单例。

3. 实现 `requestAllFilesPermission(activity: Activity)`，处理 Intent 跳转。
**提示**：
别忘了在 AndroidManifest.xml 里添加 `<uses-permission android:name="android.permission.MANAGE_EXTERNAL_STORAGE" />`。

#### Task 2 · 制造垃圾 (The Litterer) ★★
**目标**：为了测试清理功能，你需要先制造一些“垃圾”。
**你需要做的事**：
1. 编写一个函数，在 `Environment.getExternalStoragePublicDirectory(Environment.DIRECTORY_DOWNLOADS)` 下创建 5 个名为 `temp_test_1.tmp` 到 `temp_test_5.tmp` 的空文件。
2. 注意：如果你 targetSDK >= 30，创建这些文件可能不需要特殊权限（因为是你自己创建的）。

#### Task 3 · 强力清洁 (The Cleaner) ★★★★
**目标**：使用 `File` API 扫描并删除垃圾文件。
**你需要做的事**：
1. 确保已获得 `MANAGE_EXTERNAL_STORAGE` 权限。
2. 遍历 Downloads 文件夹。
3. 找出所有以 `.tmp` 结尾 **且** 文件名包含 `temp_test` 的文件。
4. 调用 `file.delete()` 删除它们，并统计删除数量。
5. 在界面上显示：“已清理 X 个垃圾文件”。
**注意**：为了安全，**千万限制删除的条件（文件名特征）**，不要误删了你自己真实下载的文件！

#### 参考实现要点

1.  **权限申请的最后手段**：始终优先使用 MediaStore 或 Storage Access Framework (SAF)。只有当应用的核心功能（如杀毒、备份、文件管理）确实无法通过这些 API 实现时，才申请 `MANAGE_EXTERNAL_STORAGE`。
2.  **优雅降级**：如果用户拒绝授权，应用不应直接崩溃或退出，而应提供基础功能（如只管理应用私有目录）。
3.  **版本适配**：使用 `Environment.isExternalStorageManager()` 前务必检查 SDK 版本 (`Build.VERSION.SDK_INT >= 30`)，旧版本需走旧的权限申请流程。
4.  **安全删除**：在使用 `File.delete()` 批量删除文件时，务必进行严格的文件名和路径校验，防止误删用户重要数据。
5.  **隐私透明**：在申请权限的界面，向用户清楚解释“为什么需要这个权限”，能显著提高授权成功率。

> 💡 **学习建议**：所有的权力都伴随着责任。在这一章，你获得了一把能打开所有门的钥匙，但请记住 Android 的设计哲学是“最小权限原则”。只有在万不得已时才使用它。对于大多数日常开发，1.3 节的 Shared Storage 和 SAF 才是你的常规武器。
