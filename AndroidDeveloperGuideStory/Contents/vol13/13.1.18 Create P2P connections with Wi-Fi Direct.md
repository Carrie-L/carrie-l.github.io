# 13.1.18 Create P2P connections with Wi-Fi Direct

夜风把营灯的火苗吹得轻轻摇晃，在草地上投下摇曳的光影。

洛芙刚把昨天写的 NSD 代码从头到尾又看了一遍，正准备关掉平板休息——"等等。"希尔忽然说。

她从笔记本后面绕过来，蹲到洛芙旁边，指着屏幕上那行代码："你这 NSD 发现，是不是必须得有一台 Wi-Fi 路由器？"

洛芙愣了一下，低头看了看代码，又抬头看着希尔："对呀……我们借的那台路由器还在营地管理站呢。"

"那如果明天我们去山里徒步，"希尔慢慢地说，"没有路由器，没有 Wi-Fi，没有任何网络基础设施——两台手机还能互相发现、连起来聊天吗？"

洛芙的眼睛慢慢变大了。

"这……不能吗？"

"能的。"黛琳不知道什么时候已经从帐篷里钻了出来，怀里抱着她那台小笔记本电脑，"这就是 Wi-Fi Direct 的主场了。"

伊莎也凑了过来，手里还端着一杯温热的柠檬水："Wi-Fi Direct……我好像在手机设置里见过这个名字。"

"当然见过。"黛琳把笔记本放在折叠桌上，屏幕亮起来，上面是 Android 开发者网站的文档页面，"Wi-Fi Direct 也叫 Wi-Fi P2P，是 Wi-Fi 联盟定义的一套标准——让设备之间可以直接互连，不需要经过 Wi-Fi 接入点。"

"就像……"希尔想了想，"就像两个人不用对讲机基地台，直接用手持对讲机就能通话？"

"差不多是这个意思。"黛琳点点头，"传统的 Wi-Fi 设备必须先连上路由器，才能互相通信。但 Wi-Fi Direct 设备可以自己充当热点，自己充当客户端，自己和自己连——两台设备碰在一起，自己就能组网。"

洛芙张了张嘴："可是这样不会互相干扰吗？没有协调者的话……"

"Wi-Fi Direct 有一个协商机制。"黛琳说，"两台设备要建立连接之前，会先互相发现，然后协商谁当 Group Owner——你可以把它理解成临时的'虚拟路由器'。这个协商是自动完成的，应用不需要操心细节。"

"那……怎么用代码实现？"洛芙问。

希尔把笔记本转过来，屏幕对着大家："这就是今天晚上的内容了。"

她顿了顿，然后说："首先，你需要三样东西——WifiP2pManager、Channel、和 BroadcastReceiver。"

"看这张图。"希尔指着屏幕，"第一步，从系统服务里拿到 WifiP2pManager——跟 WifiManager 的获取方式一样，从 Context.getSystemService 拿。"

"第二步，调用 initialize() 创建一个 Channel。"希尔说，"这个 Channel 很重要——它代表了你的 App 和 Wi-Fi P2P Framework 之间的一个会话连接。之后所有 P2P 操作都要带着这个 Channel。"

"就像……NSD 里的 NsdManager？"洛芙问。

"思路有点像，但更严格。"黛琳说，"NSD 的 registerService() 只需要传一个 listener。但 WifiP2pManager 的每个操作都需要传 Channel——没有 Channel，P2P Framework 根本不知道是哪 App 在请求。"

"第三步，注册 BroadcastReceiver 监听 P2P 相关的广播。"希尔说，"跟 NSD 一样——Wi-Fi Direct 也是事件驱动的，状态变化通过广播通知你。"

"那这三种东西怎么串起来？"洛芙问。

希尔在键盘上敲了几行代码：

```kotlin
// WifiP2pConnector.kt
// Wi-Fi Direct P2P 连接管理器
// 依赖:android.net.wifi.p2p.WifiP2pManager, android.content.Context

class WifiP2pConnector(private val context: Context) {

    // 第一步:从系统服务获取 WifiP2pManager
    private val wifiP2pManager: WifiP2pManager? by lazy {
        context.getSystemService(Context.WIFI_P2P_SERVICE) as? WifiP2pManager
    }

    // 第二步:初始化 Channel(代表 App 与 P2P Framework 的会话)
    // Channel 必须在主线程创建
    private var channel: WifiP2pManager.Channel? = null

    // 第三步:BroadcastReceiver 用于接收 P2P 状态变化的广播
    private var broadcastReceiver: BroadcastReceiver? = null

    // 第四步:IntentFilter 过滤我们需要的广播
    private val intentFilter = IntentFilter().apply {
        addAction(WifiP2pManager.WIFI_P2P_STATE_CHANGED_ACTION)       // P2P 是否启用
        addAction(WifiP2pManager.WIFI_P2P_PEERS_CHANGED_ACTION)     // 设备列表变化
        addAction(WifiP2pManager.WIFI_P2P_CONNECTION_CHANGED_ACTION) // 连接状态变化
        addAction(WifiP2pManager.WIFI_P2P_THIS_DEVICE_CHANGED_ACTION) // 本设备信息变化
    }

    // 初始化方法
    fun initialize() {
        channel = wifiP2pManager?.initialize(
            context,                              // Context
            Looper.getMainLooper(),               // 必须传主线程 Looper
            object : WifiP2pManager.ChannelCallback {
                override fun onChannelConnected(networkId: Int) {
                    // Channel 连接成功,可以开始 P2P 操作了
                    Log.d("WiFi-P2P", "Channel 已连接, networkId: $networkId")
                }

                override fun onChannelDisconnected() {
                    // Channel 断开,通常是 P2P Framework 重置了
                    Log.w("WiFi-P2P", "Channel 断开,需要重新 initialize()")
                    // 这里可以尝试重新初始化
                }
            }
        )
        Log.d("WiFi-P2P", "WifiP2pManager 初始化完成")
    }

    // 注册广播接收器(通常在 onResume 中调用)
    fun registerReceiver() {
        broadcastReceiver = WifiP2pBroadcastReceiver(wifiP2pManager, channel!!)
        context.registerReceiver(broadcastReceiver, intentFilter)
        Log.d("WiFi-P2P", "广播接收器已注册")
    }

    // 注销广播接收器(通常在 onPause 中调用)
    fun unregisterReceiver() {
        broadcastReceiver?.let {
            context.unregisterReceiver(it)
            broadcastReceiver = null
            Log.d("WiFi-P2P", "广播接收器已注销")
        }
    }
}
```

"initialize() 要传主线程的 Looper——这是 Android 的硬性要求。"希尔强调，"因为 ChannelCallback 的回调是在主线程回调的。"

"为什么要主动注销 BroadcastReceiver？"洛芙问。

"因为 P2P 广播是高频事件。"黛琳说，"如果应用切到后台了还继续接收 P2P 广播，会消耗不必要的资源。而且 Context.unregisterReceiver() 和 registerReceiver() 必须配对——漏掉任何一个都会出问题。"

"那在 onResume 和 onPause 里配对注册和注销？"洛芙飞快地记。

"对，这跟 NSD 的 onCreate/onDestroy 配对不一样——P2P 更适合用 onResume/onPause，因为用户能看到这个 Activity 的时候才需要 P2P，看不到的时候就不需要。"黛琳说。

希尔把屏幕往下滑，显示下一段代码。

"现在来看核心功能——发现附近设备。"她说，"调用 discoverPeers() 就行了。"

```kotlin
    // 开始发现附近的支持 Wi-Fi Direct 的设备
    fun discoverPeers() {
        channel?.let { ch ->
            wifiP2pManager?.discoverPeers(
                ch,                                  // Channel
                object : WifiP2pManager.ActionListener {
                    override fun onSuccess() {
                        // 发现操作发起成功(不表示发现了设备!)
                        // 结果通过广播 WIFI_P2P_PEERS_CHANGED_ACTION 返回
                        Log.d("WiFi-P2P", "discoverPeers() 调用成功,正在扫描...")
                    }

                    override fun onFailure(reason: Int) {
                        // 发现失败
                        // reason 可能是:
                        //   WifiP2pManager.P2P_NOT_SUPPORTED - 设备不支持 P2P
                        //   WifiP2pManager.BUSY - P2P 操作过于频繁
                        //   WifiP2pManager.ERROR - 其他错误
                        Log.e("WiFi-P2P", "discoverPeers() 失败,reason: $reason")
                    }
                }
            )
        } ?: Log.w("WiFi-P2P", "Channel 未初始化,无法发现设备")
    }

    // 停止发现
    fun stopDiscovery() {
        channel?.let { ch ->
            wifiP2pManager?.stopPeerDiscovery(
                ch,
                object : WifiP2pManager.ActionListener {
                    override fun onSuccess() {
                        Log.d("WiFi-P2P", "停止发现成功")
                    }

                    override fun onFailure(reason: Int) {
                        Log.e("WiFi-P2P", "停止发现失败,reason: $reason")
                    }
                }
            )
        }
    }
```

"等一下，"洛芙举起手，"onSuccess 只表示'发起发现成功'，不表示'发现了设备'？"

"没错。"希尔说，"这个跟 NSD 的 discoverServices() 一样——API 调用成功只是说'系统开始干活了'，真正的结果是通过广播返回的。当发现了设备列表之后，系统会发 WIFI_P2P_PEERS_CHANGED_ACTION 这个广播，你收到之后调用 requestPeers() 去拿设备列表。"

"就像 NSD 里的 onServiceFound 一样？"洛芙说。

"对，但是细节不一样——NSD 的 onServiceFound 是直接回调给你的。但 Wi-Fi Direct 的设备列表是通过另一个 API requestPeers() 拿的，不是直接传给你的。"

黛琳在旁边补充道："这是 Android Wi-Fi P2P API 设计上的一个小坑，很多新手会卡在这里——收到 PEERS_CHANGED 广播之后，不知道下一步该干什么。"

"下一步就是调用 requestPeers()！"希尔说。

她在 BroadcastReceiver 的实现里敲了下去：

```kotlin
// Wi-Fi Direct 专用的 BroadcastReceiver
class WifiP2pBroadcastReceiver(
    private val manager: WifiP2pManager?,
    private val channel: WifiP2pManager.Channel?
) : BroadcastReceiver() {

    // 对外的回调接口(供 Activity 或 ViewModel 使用)
    var onPeersAvailable: ((List<WifiP2pDevice>) -> Unit)? = null
    var onConnectionAvailable: ((WifiP2pInfo) -> Unit)? = null
    var onP2pStateEnabled: ((Boolean) -> Unit)? = null

    override fun onReceive(context: Context, intent: Intent) {
        when (intent.action) {

            // P2P 状态是否启用(Wi-Fi Direct 开关)
            WifiP2pManager.WIFI_P2P_STATE_CHANGED_ACTION -> {
                val state = intent.getIntExtra(
                    WifiP2pManager.EXTRA_WIFI_STATE,
                    WifiP2pManager.WIFI_P2P_STATE_DISABLED
                )
                val isEnabled = state == WifiP2pManager.WIFI_P2P_STATE_ENABLED
                Log.d("WiFi-P2P", "P2P 状态:${if (isEnabled) "已启用" else "已禁用"}")
                onP2pStateEnabled?.invoke(isEnabled)
            }

            // 设备列表发生了变化(发现了新设备,或设备消失了)
            WifiP2pManager.WIFI_P2P_PEERS_CHANGED_ACTION -> {
                Log.d("WiFi-P2P", "设备列表变化,请求获取设备列表...")
                // 收到此广播后,必须调用 requestPeers() 获取当前设备列表
                requestPeersList()
            }

            // 连接状态发生了变化(连接上了,或断开了)
            WifiP2pManager.WIFI_P2P_CONNECTION_CHANGED_ACTION -> {
                Log.d("WiFi-P2P", "连接状态变化,请求获取连接信息...")
                // 收到此广播后,调用 requestConnectionInfo() 获取连接详情
                requestConnectionInfo()
            }

            // 本设备自身信息变化(设备名、状态等)
            WifiP2pManager.WIFI_P2P_THIS_DEVICE_CHANGED_ACTION -> {
                val device = intent.getParcelableExtra<WifiP2pDevice>(
                    WifiP2pManager.EXTRA_WIFI_P2P_DEVICE
                )
                Log.d("WiFi-P2P", "本设备信息变化: ${device?.deviceName} (${device?.deviceAddress})")
            }
        }
    }

    // 从 P2P Framework 请求当前发现的设备列表
    private fun requestPeersList() {
        manager?.requestPeers(channel, object : WifiP2pManager.PeerListListener {
            override fun onPeersAvailable(peers: WifiP2pDeviceList) {
                val deviceList = peers.deviceList.toList()
                Log.d("WiFi-P2P", "发现 ${deviceList.size} 台设备:")
                deviceList.forEach { device ->
                    Log.d("WiFi-P2P", "  - ${device.deviceName} (${device.deviceAddress}) " +
                            "状态:${getDeviceStatusString(device.status)}")
                }
                // 回调给 Activity/ViewModel 更新 UI
                onPeersAvailable?.invoke(deviceList)
            }
        })
    }

    // 请求当前连接的详细信息(包含 Group Owner 信息)
    private fun requestConnectionInfo() {
        manager?.requestConnectionInfo(channel, object : WifiP2pManager.ConnectionInfoListener {
            override fun onConnectionInfoAvailable(info: WifiP2pInfo) {
                // info 包含:groupOwnerAddress(P2P 连接的对端地址)
                //          isGroupOwner(本设备是否是 Group Owner)
                //          groupFormed(是否已形成 P2P Group)
                Log.d("WiFi-P2P", """
                    连接信息:
                    - Group 已形成:${info.groupFormed}
                    - 本设备是 GroupOwner:${info.isGroupOwner}
                    - GroupOwner 地址:${info.groupOwnerAddress?.hostAddress}
                """.trimIndent())
                onConnectionAvailable?.invoke(info)
            }
        })
    }

    // 设备状态码转可读字符串
    private fun getDeviceStatusString(status: Int): String = when (status) {
        WifiP2pDevice.AVAILABLE -> "可用"
        WifiP2pDevice.INVITED -> "正在连接"
        WifiP2pDevice.CONNECTED -> "已连接"
        WifiP2pDevice.FAILED -> "失败"
        WifiP2pDevice.UNAVAILABLE -> "不可用"
        else -> "未知($status)"
    }
}
```

希尔往椅背上一靠，指着代码说："这就是 Wi-Fi Direct 的发现流程——"

"三步走。"洛芙数着手指，"第一步调用 discoverPeers()，第二步等 PEERS_CHANGED 广播，第三步调用 requestPeers() 拿列表。"

"没错。"希尔说，"而且这个流程可以重复——discoverPeers() 可以一直开着，系统会持续扫描。每发现新设备就会发广播。"

"跟 NSD 的 discoverServices() 一样持续监听。"洛芙说。

"对。不过有一点区别——"黛琳说，"NSD 的 onServiceFound 会直接把服务信息给你。但 Wi-Fi Direct 的 requestPeers() 拿到的是 WifiP2pDevice 对象，里面有设备名、设备地址、状态，但没有 IP 地址。"

"那 IP 地址什么时候才能拿到？"洛芙问。

"建立 P2P 连接之后。"黛琳说，"当你调用 connect() 成功建立了 P2P Group，并且收到 CONNECTION_CHANGED 广播之后，调用 requestConnectionInfo() ——这时候拿到的 WifiP2pInfo 里才有 groupOwnerAddress。这个 IP 是 P2P Group 内部的 IP，跟普通 Wi-Fi 的 IP 段不一样。"

"所以 NSD 是在同一个 Wi-Fi 局域网里发现服务，"洛芙慢慢地说，"Wi-Fi Direct 是不需要 Wi-Fi 局域网，自己设备之间就能组一个临时局域网……"

"而且这个临时局域网的 IP 是内部分配的，不是 DHCP 租约那种。"希尔补充道。

伊莎端着柠檬水在旁边听着，忽然说："就像我们露营的时候——NSD 是大家都在营地管理处的 Wi-Fi 覆盖范围内，用萤火虫找彼此。但 Wi-Fi Direct 是大家都不带对讲机基地台，直接用手持对讲机，自己就能开一个临时频道。"

"这个比喻很准确。"黛琳点点头。

```kotlin
    // 连接到指定的 P2P 设备
    fun connectToDevice(targetDevice: WifiP2pDevice) {
        // 创建连接配置
        val config = WifiP2pConfig().apply {
            deviceAddress = targetDevice.deviceAddress   // 必填:目标设备的 MAC 地址
            // wps.setup 决定 Group Owner 协商时的 WPS 方式
            // WpsInfo.PBC = 按下按钮配对(最简单)
            // WpsInfo.DISPLAY = 显示 PIN 码配对
            wps.setup = WpsInfo.PBC
        }

        channel?.let { ch ->
            wifiP2pManager?.connect(
                ch,
                config,
                object : WifiP2pManager.ActionListener {
                    override fun onSuccess() {
                        // 连接请求发起成功!
                        // 结果通过广播 WIFI_P2P_CONNECTION_CHANGED_ACTION 返回
                        Log.d("WiFi-P2P", "连接请求已发送至 ${targetDevice.deviceName}")
                    }

                    override fun onFailure(reason: Int) {
                        Log.e("WiFi-P2P", "连接请求失败,reason: $reason")
                    }
                }
            )
        }
    }

    // 移除当前 P2P 连接(断开连接)
    fun removeGroup() {
        channel?.let { ch ->
            wifiP2pManager?.removeGroup(
                ch,
                object : WifiP2pManager.ActionListener {
                    override fun onSuccess() {
                        Log.d("WiFi-P2P", "已移除 P2P Group,连接断开")
                    }

                    override fun onFailure(reason: Int) {
                        Log.e("WiFi-P2P", "移除 Group 失败,reason: $reason")
                    }
                }
            )
        }
    }
```

"connect() 之后，系统会做 Group Owner 协商。"希尔说，"两台设备会交换信息，决定谁当 Group Owner——Group Owner 相当于临时的虚拟路由器，所有流量都要经过它。"

"怎么决定谁当 Group Owner？"洛芙问。

"应用层不需要决定，这是 Framework 自动协商的。"黛琳说，"通常是谁的权重更高谁当 Owner。但应用可以设置一个 hint——如果你想强制自己当 Group Owner，可以设置 wps.setup = WpsInfo.DISPLAY，然后设置 wps.devicePassword = 一个值。不过这对大多数应用来说太复杂了。"

"Group Owner 协商完成之后，系统会发 WIFI_P2P_CONNECTION_CHANGED_ACTION 广播。"希尔说，"你收到广播之后，调用 requestConnectionInfo()，就能拿到 WifiP2pInfo——里面有 isGroupOwner 和 groupOwnerAddress。"

"然后呢？"洛芙问。

"然后，你作为应用层，拿到这个信息之后，就可以建立 Socket 连接了——跟 NSD 一样，拿到 IP 和端口之后，用普通的 Socket API 去通信。"

"等一下！"洛芙忽然说，"那和 NSD 的区别到底是什么？"

希尔和黛琳对视了一眼。

"区别在于发现和连接的基础设施。"黛琳慢慢地说。

```kotlin
// P2pMainActivity.kt
// Wi-Fi Direct P2P 完整示例
// 演示:初始化 → 发现设备 → 连接 → Socket 通信

class P2pMainActivity : AppCompatActivity() {

    private lateinit var connector: WifiP2pConnector
    private var peers = mutableListOf<WifiP2pDevice>()
    private var connectionInfo: WifiP2pInfo? = null
    private var isGroupOwner = false

    private val binding: ActivityP2pMainBinding by lazy {
        ActivityP2pMainBinding.inflate(layoutInflater)
    }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(binding.root)

        // 初始化 P2P 连接管理器
        connector = WifiP2pConnector(this)

        // 设置按钮点击
        binding.btnDiscover.setOnClickListener {
            connector.discoverPeers()
        }

        binding.btnConnect.setOnClickListener {
            // 连接列表中选中的第一个设备
            val selectedDevice = getSelectedDevice()
            selectedDevice?.let { connector.connectToDevice(it) }
        }

        binding.btnDisconnect.setOnClickListener {
            connector.removeGroup()
        }
    }

    override fun onResume() {
        super.onResume()
        connector.initialize()
        connector.registerReceiver()
        
        // 设置广播回调
        connector.broadcastReceiver?.apply {
            onPeersAvailable = { deviceList ->
                peers.clear()
                peers.addAll(deviceList)
                runOnUiThread { updatePeerListUI(deviceList) }
            }
            onConnectionAvailable = { info ->
                connectionInfo = info
                isGroupOwner = info.isGroupOwner
                runOnUiThread { updateConnectionUI(info) }
                // 根据角色决定行为
                if (info.groupFormed) {
                    if (isGroupOwner) {
                        // 我是 Group Owner:启动 ServerSocket 等待连接
                        startServerSocket()
                    } else {
                        // 我是 Client:主动连接 Group Owner
                        info.groupOwnerAddress?.hostAddress?.let { goIP ->
                            startClientConnection(goIP)
                        }
                    }
                }
            }
            onP2pStateEnabled = { enabled ->
                Log.d("P2P", "Wi-Fi Direct ${if (enabled) "可用" else "不可用"}")
            }
        }
    }

    override fun onPause() {
        super.onPause()
        connector.unregisterReceiver()
    }

    private fun getSelectedDevice(): WifiP2pDevice? {
        // 从 UI RecyclerView 中获取选中项
        // 此处省略 UI 绑定代码,假设有选中逻辑
        return peers.firstOrNull()
    }

    // 反模式:在主线程执行 Socket 操作
    // 【错误示例】—— 绝对不要这样做!
    private fun badSocketExample() {
        val socket = Socket(connectionInfo!!.groupOwnerAddress!!.hostAddress, 8888)
        // ^^^^^^^^ NetworkOnMainThreadException! Android 从 Honeycomb 开始
        // 就禁止在主线程执行网络操作了,这里会直接崩溃
        val writer = PrintWriter(socket.getOutputStream(), true)
        writer.println("Hello!")
    }

    // 正确的做法:在后台线程执行 Socket I/O
    private fun startServerSocket() {
        Thread {
            try {
                val serverSocket = ServerSocket(8888)
                Log.d("P2P-Server", "ServerSocket 监听端口 8888")
                val clientSocket = serverSocket.accept()  // 阻塞等待
                handleConnection(clientSocket)
                serverSocket.close()
            } catch (e: Exception) {
                Log.e("P2P-Server", "ServerSocket 错误:${e.message}")
            }
        }.start()
    }

    private fun startClientConnection(serverIP: String) {
        Thread {
            try {
                val socket = Socket(serverIP, 8888)
                Log.d("P2P-Client", "已连接到 $serverIP:8888")
                val writer = PrintWriter(
                    OutputStreamWriter(socket.getOutputStream(), StandardCharsets.UTF_8),
                    true
                )
                writer.println("Hello from P2P Client!")
                socket.close()
            } catch (e: Exception) {
                Log.e("P2P-Client", "连接失败:${e.message}")
            }
        }.start()
    }

    private fun handleConnection(clientSocket: Socket) {
        try {
            val reader = BufferedReader(
                InputStreamReader(clientSocket.getInputStream(), StandardCharsets.UTF_8)
            )
            val message = reader.readLine()
            Log.d("P2P-Server", "收到消息:$message")
            // 可以在这里回复
            val writer = PrintWriter(clientSocket.getOutputStream(), true)
            writer.println("Received: $message")
            reader.close()
            writer.close()
            clientSocket.close()
        } catch (e: Exception) {
            Log.e("P2P-Server", "处理连接时出错:${e.message}")
        }
    }

    private fun updatePeerListUI(devices: List<WifiP2pDevice>) {
        // 更新 RecyclerView 显示设备列表
        binding.tvStatus.text = "发现 ${devices.size} 台设备"
    }

    private fun updateConnectionUI(info: WifiP2pInfo) {
        val role = if (isGroupOwner) "Group Owner(服务端)" else "Client(客户端)"
        binding.tvStatus.text = "已连接! 角色:$role"
    }
}
```

## 专业技术总结

**Wi-Fi Direct (Wi-Fi P2P)** -- Android 提供的点对点无线直连技术，允许设备在不依赖 Wi-Fi 接入点的情况下直接互联。Wi-Fi Direct 通过设备间协商产生 Group Owner，形成临时 P2P 网络，连接建立后使用标准 Socket API 进行通信。与 NSD 的本质区别在于：NSD 依赖现有局域网基础设施，Wi-Fi Direct 完全独立。

#### 反模式与陷阱

1. **收到 PEERS_CHANGED 广播后不调用 requestPeers()** -- Wi-Fi Direct 的设备列表不是直接通过回调传给你的，而是需要主动调用 requestPeers() 去拉取。这是 Wi-Fi Direct API 最容易踩的坑。修复：收到 PEERS_CHANGED 广播后，立即调用 requestPeers() 获取最新设备列表。

2. **在主线程执行 Socket I/O 操作** -- 拿到 WifiP2pInfo 里的 groupOwnerAddress IP 之后，如果直接创建 Socket，Android 会抛 NetworkOnMainThreadException（从 API 11 开始）。修复：所有 Socket 操作（ServerSocket、Socket、read、write）必须放在 Thread {} 或 Kotlin 协程中执行。

3. **跳过 Group Owner 协商完成的检查直接通信** -- connect() 成功后，P2P Group 的建立和 Go Negotiation 需要几秒钟。如果在 groupFormed = true 确认之前就尝试 Socket 连接，会失败。修复：必须在 ConnectionInfoListener.onConnectionInfoAvailable() 确认 groupFormed = true 之后，才开始 Socket 操作。

4. **discoverPeers() 后不等待广播，直接查设备列表** -- discoverPeers() 是异步的，发起成功后立即返回。设备列表需要等系统扫描完成之后，通过 PEERS_CHANGED 广播通知。修复：必须等待 PEERS_CHANGED 广播，再调用 requestPeers()，不能同步查询。

5. **不申请位置权限就调用 discoverPees()/connect()** -- 从 Android 8.0 开始，Wi-Fi Direct 扫描会被视为一种位置信息来源，没有位置权限时 API 调用会静默失败或返回空列表。修复：必须在调用 P2P API 前申请 ACCESS_FINE_LOCATION（Android 12 及以下）或 NEARBY_WIFI_DEVICES（Android 13+）。

6. **discoverPeers() 和 stopPeerDiscovery() 不配对** -- discoverPeers() 是持续扫描，如果不调用 stopPeerDiscovery() 就退出 Activity/Fragment，扫描会继续在后台消耗资源。修复：onPause() 或 onDestroy() 中调用 stopPeerDiscovery()，确保扫描和停止扫描配对。

7. **Channel 未初始化就调用 P2P API** -- initialize() 必须在主线程调用，且需要 WifiP2pManager.ChannelCallback 的 onChannelConnected() 回调确认连接建立。如果在 Channel 就绪前就调用 discoverPeers()/connect()，会收到 BUSY 或 ERROR 错误。修复：等 ChannelCallback.onChannelConnected() 回调确认就绪后，再发起 P2P 操作。

#### 设计哲学

Wi-Fi Direct 的设计体现了**自组织网络（Self-Organizing Network）**思想：不需要中心基础设施，设备之间直接协商，自动形成临时网络。这种模式非常适合露营、临时会议、户外活动等没有网络基础设施的场景。

核心工程原则：

1. **Channel 即会话** -- 每次 App 与 P2P Framework 的交互都需要带着 Channel，Channel 丢失（onChannelDisconnected）意味着需要重新 initialize()，不能复用旧的 Channel。
2. **发现是异步的，结果靠广播** -- discoverPeers() 发起后立即返回，真正发现设备靠 WIFI_P2P_PEERS_CHANGED_ACTION 广播，收到广播后必须主动调用 requestPeers() 获取列表。
3. **Group Owner 角色决定通信模式** -- isGroupOwner = true 时充当服务端监听端口；isGroupOwner = false 时充当客户端主动连接。两端运行同一份代码，但行为不同——这是 P2P 自适应的精髓。
4. **Socket I/O 永远在后台** -- P2P Socket 与普通 Socket 没有任何区别，都是阻塞 I/O，都必须后台执行。拿到 IP 和开始 Socket 之间没有魔法，就是普通的 Thread 或协程。
5. **P2P 发现有功耗成本** -- Wi-Fi Direct 的持续扫描会显著增加电池消耗。真实应用应该在需要时才开启发现，用户停止交互后及时停止（stopPeerDiscovery）。

#### 面试热身

**Q1:Wi-Fi Direct 和 NSD 的核心区别是什么？分别在什么场景下使用？**

Wi-Fi Direct 和 NSD 有三个核心区别：第一，NSD 依赖现有的 Wi-Fi 局域网基础设施（AP/路由器），mDNS 广播通过这个基础设施传播；Wi-Fi Direct 不需要任何 AP，设备之间直接协商建立 P2P Group。第二，NSD 发现的是"服务"（服务名+类型+端口），Wi-Fi Direct 发现的是"设备"（设备名+MAC 地址）。第三，NSD 在 resolveService() 后直接拿到 IP 和端口，Wi-Fi Direct 在 connect() 成功并收到 ConnectionInfoListener 回调后才能通过 groupOwnerAddress 拿到 P2P Group 内部的 IP。使用场景：NSD 适合在已有 Wi-Fi 基础设施下做服务发现（投屏、局域网游戏）；Wi-Fi Direct 适合在没有网络基础设施的户外、临时场景下实现设备直连（多人游戏、文件传输、应急通信）。

**Q2:Wi-Fi Direct 的 Group Owner 协商是如何工作的？应用层如何根据 isGroupOwner 决定行为？**

Group Owner 协商（P2P Go Negotiation）是 Wi-Fi Direct 设备之间自动进行的过程，两台设备交换各自的 Group Owner 权重（Intent 值），权重高的成为 Group Owner，相当于临时的虚拟路由器。所有 P2P 流量都经过 Group Owner 转发。应用层不需要干预协商过程，只需要在收到 ConnectionInfoListener.onConnectionInfoAvailable() 回调后，检查 isGroupOwner 的值：如果为 true，说明本设备是 Group Owner，应当启动 ServerSocket 监听端口等待对方连接；如果为 false，说明对方是 Group Owner，应当主动连接 groupOwnerAddress 的 IP。这使得同一份代码在两台设备上可以有不同的行为——这就是 P2P 角色的自适应。

**Q3:Wi-Fi Direct 的发现流程是怎样的？为什么收到 PEERS_CHANGED 广播后必须调用 requestPeers()？**

Wi-Fi Direct 的发现流程是：首先调用 discoverPeers() 发起扫描（立即返回），然后设备之间通过 P2P Probe Request/Response 交换信息，扫描到设备后系统发送 WIFI_P2P_PEERS_CHANGED_ACTION 广播。收到广播后，必须调用 requestPeers() 主动拉取设备列表——这个列表不是自动推送给你的，而是需要你主动请求。调用 requestPeers() 时，系统调用 PeerListListener.onPeersAvailable() 回调，在这个回调里才能拿到 List<WifiP2pDevice>。如果不调用 requestPeers()，即使收到了广播，也不知道发现了哪些设备。

**Q4:Wi-Fi Direct 开发中有哪些常见的性能陷阱和功耗问题？**

Wi-Fi Direct 的主要性能陷阱包括：discoverPeers() 是持续扫描，发现过程通常需要几秒到十几秒，不能期望像 NSD 那样快速响应；Wi-Fi Direct 扫描期间设备会切换信道并持续发送 Probe Request，功耗显著增加，必须在不需要时调用 stopPeerDiscovery() 停止扫描；Go Negotiation 过程可能需要 1-3 秒，应用层需要给协商足够的时间，不能超时后立刻重试。功耗优化建议：只在用户明确需要发现设备时才开启扫描；设置合理的扫描超时时间（比如 30 秒后自动停止）；设备进入省电模式时停止扫描；在不需要 P2P 时及时 removeGroup() 清理 P2P Group。

**Q5:Wi-Fi Direct 的权限模型是什么？为什么需要位置权限？**

Wi-Fi Direct 的权限包括：ACCESS_WIFI_STATE 和 CHANGE_WIFI_STATE 用于基本 P2P 操作，ACCESS_FINE_LOCATION（Android 12 及以下）或 NEARBY_WIFI_DEVICES（Android 13+）用于发现附近设备，INTERNET 用于 Socket 通信。需要位置权限的原因是：从 Android 8.0（API 26）开始，Wi-Fi Direct 扫描被视为一种位置信息来源——通过分析周围的 Wi-Fi P2P 信号特征（包括信号强度、信道信息等），可以推断出设备的大致物理位置，这属于一种侧信道定位能力。因此 Google 要求必须获得位置权限才能发现附近的 P2P 设备，目的是防止恶意应用在用户不知情的情况下通过 P2P 扫描追踪用户位置。此外，应用应当在位置权限被撤销时优雅降级，提示用户需要权限才能使用 P2P 功能。

#### 参考实现要点

1. **Wi-Fi Direct 和 NSD 共享相同的 Socket 通信层**：Wi-Fi Direct 建立 P2P Group 并获取 groupOwnerAddress IP 之后，Socket 通信代码与 NSD 完全相同。所有 Socket 操作（ServerSocket/Socket 的创建、connect、read、write）必须在 Thread {} 或协程中执行，禁止在主线程操作，否则 NetworkOnMainThreadException。

2. **Channel 的生命周期必须与 Activity/Fragment 生命周期配对**：initialize() 在 onResume() 中调用（等 ChannelCallback.onChannelConnected 确认就绪后再操作），registerReceiver() 在 onResume() 中调用，unregisterReceiver() 在 onPause() 中调用。discoverPeers() 和 stopPeerDiscovery() 也要配对：发现按钮触发 discoverPeers()，停止按钮或 onPause() 中调用 stopPeerDiscovery()。

3. **P2P 发现结果依赖广播和主动拉取**：`discoverPeers()` 成功后返回 actionListener.onSuccess()，但不包含任何设备信息。真正的设备列表通过 WIFI_P2P_PEERS_CHANGED_ACTION 广播通知，收到广播后必须调用 `requestPeers(Channel, PeerListListener)` 主动拉取。`connect()` 成功同理，连接信息通过 WIFI_P2P_CONNECTION_CHANGED_ACTION 广播通知，收到后调用 `requestConnectionInfo(Channel, ConnectionInfoListener)` 拉取 WifiP2pInfo。

4. **权限申请必须先于 P2P API 调用**：从 Android 8.0 开始，Wi-Fi Direct API 需要位置权限才能正常工作。在调用 `discoverPeers()`、`connect()`、`requestPeers()` 等任何 P2P API 之前，必须确保已获得 `ACCESS_FINE_LOCATION`（API ≤ 33）或 `NEARBY_WIFI_DEVICES`（API 34+）权限。建议使用 `ActivityResultContracts.RequestMultiplePermissions` 或 `ActivityResultContracts.RequestPermission` 管理权限请求生命周期。

5. **Group Owner 角色的自适应是 Wi-Fi Direct 编程的核心模式**：同一份代码在 isGroupOwner = true 时启动 ServerSocket 等连接，在 isGroupOwner = false 时主动连接 groupOwnerAddress。这意味着 Group Owner 充当"虚拟路由器 + 服务端"，Client 充当"客户端"。Socket 通信建立后，双方完全对等——这就是 P2P 的精髓。

## 今日关键词

**WifiP2pManager** -- Android 系统服务，Wi-Fi Direct 的主入口，提供 initialize()、discoverPeers()、connect()、removeGroup()、requestPeers()、requestConnectionInfo() 等方法。

**WifiP2pManager.Channel** -- App 与 Wi-Fi P2P Framework 之间的会话通道，通过 initialize() 创建，每个 P2P 操作都需要携带此 Channel。

**WifiP2pDevice** -- 代表一个发现的 P2P 设备，包含 deviceName（设备名）、deviceAddress（MAC 地址，用于 connect()）、status（设备状态：AVAILABLE/INVITED/CONNECTED/FAILED/UNAVAILABLE）。

**WifiP2pConfig** -- 连接配置类，包含 deviceAddress（目标设备 MAC 地址）和 wps（WPS 配对方式，通常设为 WpsInfo.PBC）。

**WifiP2pInfo** -- 连接详情类，包含 groupFormed（Group 是否已形成）、isGroupOwner（本设备是否为 Group Owner）、groupOwnerAddress（Group Owner 的 IP 地址）。

**WifiP2pDeviceList** -- 发现的设备列表容器，通过 requestPeers() 获取，内含 WifiP2pDevice 集合。

**WIFI_P2P_PEERS_CHANGED_ACTION** -- P2P 广播：设备列表发生变化（发现了新设备或设备消失），收到后必须调用 requestPeers() 获取最新列表。

**WIFI_P2P_CONNECTION_CHANGED_ACTION** -- P2P 广播：P2P 连接状态变化（连接成功或断开），收到后必须调用 requestConnectionInfo() 获取连接详情。

**WIFI_P2P_STATE_CHANGED_ACTION** -- P2P 广播：Wi-Fi Direct 功能是否启用（系统设置中的 P2P 开关），state = WIFI_P2P_STATE_ENABLED 表示可用。

**Group Owner** -- P2P Group 中的临时"虚拟路由器"，由两台设备协商产生。所有 P2P 流量经过 Group Owner 转发。isGroupOwner = true 的设备应启动 ServerSocket。

**Go Negotiation** -- Group Owner 协商过程，两台设备自动交换权重（Intent），权重高的成为 Group Owner。应用层不需要干预，结果通过 ConnectionInfoListener.onConnectionInfoAvailable() 返回。

**ChannelCallback** -- initialize() 时传入的回调接口，包含 onChannelConnected()（Channel 就绪）和 onChannelDisconnected()（Channel 断开，需要重新 initialize）。

**ActionListener** -- discoverPeers()、connect()、removeGroup() 等操作的通用回调接口，包含 onSuccess() 和 onFailure()，但注意：onSuccess 不包含操作结果，结果靠广播返回。

**PeerListListener** -- requestPeers() 的回调接口，onPeersAvailable() 返回当前发现的设备列表 List<WifiP2pDevice>。

**ConnectionInfoListener** -- requestConnectionInfo() 的回调接口，onConnectionInfoAvailable() 返回 WifiP2pInfo（包含 Group Owner 角色和 IP）。

**NEARBY_WIFI_DEVICES 权限** -- Android 13（API 33）引入的附近设备权限，用于发现 Wi-Fi Direct 设备。从 API 34 开始强制要求，与 ACCESS_FINE_LOCATION 二选一（或同时需要，取决于 OEM 实现）。

**WpsInfo** -- WPS（Wi-Fi Protected Setup）配对信息类，用于设置 P2P 连接时的配对方式。WpsInfo.PBC（按下按钮配对）是最常用的简单方式。
