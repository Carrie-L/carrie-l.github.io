# 执行

午后的阳光渐渐偏转，湖面的波光从耀眼变得柔和。洛芙把笔记本合上，伸了个懒腰，方才学习仿真器控制的兴奋感慢慢沉淀下来，取而代之的是一种满足后的宁静。

“接下来干什么？”希尔把手机收起来， efficacy 般的目光投向黛琳。

黛琳没有立即回答。她从背包里翻出一块旧的——确切地说，是从背包里掏出一个灰白色的U盘。U盘的塑料壳已经有些磨损，边角露出内部的金属触点。

“这个呢？”她把U盘在指尖转了两圈，“昨天我们看了仿真器，今天看看怎么‘执行’。”

伊莎正在整理睡袋，听到这话好奇地抬起头：“执行？是说运行代码吗？”

“是，也不是。”黛琳微微一笑，把U盘放在草地上，“运行代码是结果，但'执行'——Execution，指的是Android Gradle插件如何安排和运行我们的构建任务。这个接口会告诉构建系统：什么时候该干什么，怎么干，要不要并行，要不要缓存。”

洛芙重新打开笔记本：“听起来像......指挥家？”

“没错。”黛琳赞许地点点头，“想象一个管弦乐队——有吹小号的，有拉提琴的，有打鼓的。Execution就是那个指挥家，决定谁先上场，谁和谁一起上，什么时候该停，什么时候该重来。”

希尔兴奋地搓了搓手：“这个我懂！就是Task的调度嘛。对吧，黛琳？”

“对，但不止于此。”黛琳把U盘捡起来，“先坐好，我们从最基础的说起——Android项目的构建，本质上就是一堆Task的执行。Execution接口，就是用来配置这些Task行为的地方。”

她顿了顿，看向三人：“你们在build.gradle里用过repackagedex()吗？或者ndkBuild()？那些配置的背后，就是Execution在起作用。”

洛芙举手：“我我我！我在上一个项目里用过ndkBuild！当时配置了abiFilters什么的。”

“那你记得是在哪个block里配置的吗？”

洛芙努力回想：“好像是......android {} 里面？”

“对，也不对。”黛琳笑着摇头，“在android {} 块里，你会看到defaultConfig、buildTypes、productFlavors——这些都是配置。但具体到'怎么执行NDK构建'这个动作，是在externalNativeBuild这个配置块里，而负责这个部分的对象，就是Execution类型。”

她打开笔记本，在空白页上画了一个简单的层次结构：

```mermaid
graph TD
    A[android {}] --> B[defaultConfig]
    A --> C[buildTypes]
    A --> D[productFlavors]
    A --> E[externalNativeBuild]
    E --> F[ndkBuild { }]
    E --> G[cmake { }]
    F --> H[Execution配置]
    G --> I[Execution配置]
```

“NDK构建和CMake构建都遵循这个模式。”黛琳指给她们看，“execution这个概念，就隐藏在这些配置块的背后。”

伊莎若有所思：“所以......Execution就像是给构建任务贴标签？告诉它怎么个'执行'法？”

“更准确地说，是给它一套行为准则。”黛琳点头，“在AGP 9.0里，你可以配置这些Execution属性：”

她在白板上依次写下：

1. **compilerFlags** —— 传递给编译器的额外flag，比如`-Wall`、`-O3`
2. **arguments** —— 构建参数，比如`-j4`指定用4个线程
3. **cFlags / cppFlags** —— C和C++的专用编译选项
4. **defines** —— 预处理器宏定义
5. **includes / excludes** —— 包含或排除特定文件

洛芙认真地记录着：“这些看起来......好像命令行参数啊。”

“对，本质上就是那么回事。”希尔插嘴道，“你想，如果直接在终端跑ndk-build，你可以写`ndk-build -j4 V=1`，对吧？Execution就是把命令行参数翻译成Gradle DSL，让你能在build.gradle里声明式地配置。”

她说着，把笔记本转过来，手指在键盘上飞舞：

```kotlin
android {
    defaultConfig {
        ndk {
            // 这种写法是旧的、简化的
            abiFilters 'armeabi-v7a', 'arm64-v8a'
        }
    }

    externalNativeBuild {
        ndkBuild {
            path "src/main/jni/Android.mk"
            
            // Execution配置开始
            execution {
                // 并行执行，-j4
                arguments "-j4"
                
                // 详细输出
                arguments "V=1"
                
                // 额外的编译器flag
                compilerFlags "-Wall" "-Wextra" "-O3"
                
                // 预处理器定义
                defines "DEBUG_MODE=1" "ENABLE_LOG=1"
                
                // C语言专用flag
                cFlags "-std=c11"
                
                // C++专用flag
                cppFlags "-std=c++17" "-fexception"
            }
        }
    }
}
```

洛芙眼睛一亮：“这看起来清晰多了！之前我都是直接改Android.mk，都不敢动build.gradle。”

“这就是DSL的好处。”希尔解释道，“Android.mk是NDK原生的构建脚本，比较底层。execution {} 这个block让你在Gradle层面用更'现代'的方式配置构建行为，而且语法统一——不管你用ndkBuild还是cmake，都能用类似的结构。”

黛琳补充道：“还不止这些。对于CMake，Execution还能配置更多细节。比如——”

她在屏幕上又补充了一段：

```kotlin
externalNativeBuild {
    cmake {
        path "src/main/cpp/CMakeLists.txt"
        
        execution {
            // CMake专用参数
            arguments "-DCMAKE_BUILD_TYPE=Release"
            arguments "-DCMAKE_TOOLCHAIN_FILE=${ndkDir}/build/cmake/android.toolchain.cmake"
            
            // 编译器优化
            cFlags "-march=armv8-a"
            cppFlags "-march=armv8-a" "-frtti" "-fexceptions"
            
            // 链接器flag
            linkerFlags "-Wl,--gc-sections" "-Wl,-z,relro" "-Wl,-z,now"
        }
    }
}
```

伊莎轻声读出来：“linkerFlags......是在链接阶段用的，对吧？”

“没错。”黛琳点头，“链接器（Linker）是整个构建流程的最后一环——把编译好的目标文件（.o）合并成最终的二进制（.so或.apk）。linkerFlags让你控制链接行为，比如去掉未使用的代码段（--gc-sections），或者启用RELRO安全特性。”

洛芙忽然想到了什么：“那个......我之前做项目的时候，遇到过APK体积过大的问题，是不是就是这里没配置好啊？”

“很可能。”希尔接过话题，“没配置linkerFlags的Release构建，默认会保留所有符号信息和调试信息。我给你看个对比——”

她在Kotlin脚本里输入：

```kotlin
// 坏味道示例 - 未优化链接的CMake配置
cmake {
    path "src/main/cpp/CMakeLists.txt"
    // 没有execution {} 配置
}

// 重构后 - 启用链接优化
cmake {
    path "src/main/cpp/CMakeLists.txt"
    execution {
        // 发布构建加链接器优化
        arguments "-DCMAKE_BUILD_TYPE=Release"
        
        linkerFlags "-Wl,--gc-sections"      // 丢弃未使用的section
        linkerFlags "-Wl,-z,relro"          // RELRO partial
        linkerFlags "-Wl,-z,now"             // RELRO full
        linkerFlags "-Wl,--strip-all"        // Release时去掉所有符号（可选）
    }
}
```

希尔指着屏幕解释：“看，第一个例子是很多新人会写的——直接指定CMakeLists.txt，但没有优化配置。第二个做了三件事：指定Release构建、启用section gc（去掉死代码）、启用RELRO（安全加固）。”

“实际效果呢？”洛芙问。

“APR体积通常能减少15%到30%。”希尔回答，“而且安全性更高。--strip-all要谨慎用，如果是调试版本就保留符号，但生产版本一定要加。”

黛琳看向洛芙：“你现在理解Execution不是什么了？”

“它不是......不是简单的'运行代码'。”洛芙思考着，“而是一套控制构建过程如何执行的规则？包括用什么参数、怎么优化、安全怎么做......”

“对。”黛琳露出微笑，“而且Execution是可组合的。不同的buildVariant（debug、release）可以有不同的Execution配置。甚至同一个项目里，ndkBuild和cmake可以并存，只要你给它们分别配置execution {} 块。”

伊莎好奇地问：“如果我既用ndkBuild又用cmake会怎样？”

“会同时执行两个构建流程，生成两套native库。”黛琳解释，“这在迁移期很常见——老代码用Android.mk，新代码用CMake。最后打包时会合并到一个APK里。”

她站起身，拍了拍裤子上的草屑：“好了，今天的露营时间差不多了。最后总结一下——Execution就是Android Gradle插件里，专门负责'怎么执行构建任务'的DSL接口。它能配置编译器参数、链接器行为、构建并行度等等。”

洛芙飞快地在本子上记下最后几行：

```
Execution DSL
├─ 控制构建任务的执行方式
├─ 配置 compilerFlags / arguments / linkerFlags
├─ 支持 ndkBuild 和 cmake
└─ 对不同 buildVariant 可有不同配置
```

收好笔记本，洛芙看着湖面出神。傍晚的光线开始变得橙红，远处的山轮廓渐渐模糊。蝉鸣声不知什么时候淡了下去，取而代之的是风吹过柳枝的沙沙声。

“明天会学什么？”洛芙问。

黛琳把U盘的灰尘擦干净，放回背包：“Execution的'兄弟'——ExecutionProfile。”

“执行……配置文件？”伊莎眨眨眼。

“对，就像你手机里有省电模式、性能模式一样——”黛琳拉上背包拉链，“ExecutionProfile让你预先定义一套执行策略，然后切换使用。下次吧。”

洛芙点点头，把最后一口凉掉的可可喝完。三人开始收拾东西，太阳已经完全沉到了山那边，天空变成了一种温暖的紫蓝色。

> **学习建议**
> 
> Execution DSL的学习重点在于理解「构建任务的执行策略」这一概念。建议先从自己项目中的externalNativeBuild块入手，尝试添加execution {} 配置并观察构建输出的变化。 linkerFlags的优化是实际项目中经常被忽略的点，值得在练习中重点关注。

## 今日关键词

**Execution** — Android Gradle Plugin中用于配置构建任务执行行为的DSL接口，可控制编译器参数、链接器选项、构建并行度等。

**Task** — Gradle构建系统中的最小执行单元，一次构建过程由多个Task组成（如compileDebugKotlin、ndkBuild等）。

**DSL（Domain Specific Language）** — 领域特定语言，在Android开发中指build.gradle中使用的声明式配置语法。

**compilerFlags** — 传递给编译器的额外参数，如优化级别、警告级别等。

**linkerFlags** — 传递给链接器的参数，控制二进制文件的链接行为和安全特性。

**--gc-sections** — 链接器参数，丢弃未使用的代码段，减少APK体积。

**RELRO（RELocation Read-Only）** — Linux安全特性，启用后可使重定位数据变为只读，防止攻击者利用。

**ndkBuild** — Android NDK原生的构建系统，使用Android.mk和Application.mk。

**cmake** — 跨平台的构建系统，Android NDK也支持通过CMakeLists.txt配置原生代码构建。

**buildVariant** — 构建变体，通常指debug、release及其组合（如debugWithMinify、releaseWithMinify等）。

**externalNativeBuild** — Android Gradle插件中专门用于配置原生代码（ndkBuild/cmake）构建的块。
