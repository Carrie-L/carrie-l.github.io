# 21.1.150 LibraryBuildFeatures

夜色渐深，湖畔的虫鸣声也变得稀疏起来。洛芙裹紧了身上的外套，目光从跳动的火焰移到黛琳正在写的代码上。

"刚才说的 LibraryBaseFlavor 是基础，"黛琳把笔记本转过来，指着屏幕上的代码说，"但光有 flavor 可不够，我们还需要告诉构建系统：库模块本身有哪些构建特性要启用或关闭。"

"库模块的构建特性？"洛芙歪着头，"难道不是和应用模块一样的吗？"

"不一样哦。"伊莎轻轻拨了拨篝火，火星子像萤火虫一样四散开来，"库模块有它自己的开关——比如要不要生成 BuildConfig 类、要不要启用 R8 混淆、能不能添加资源值。这些都叫做 LibraryBuildFeatures。"

希尔不知道什么时候已经从背包里翻出了另一个小本子，哗啦哗啦翻到某一页："我之前做 SDK 开发的时候，被这个坑过——库模块默认不生成 BuildConfig，但我以为和 app 一样，结果调用 BuildConfig.VERSION_NAME 的时候编译直接报错。"

"那这个 LibraryBuildFeatures 到底能配置哪些东西？"洛芙往前凑了凑，好奇心完全被勾起来了。

黛琳没有直接回答，而是从地上捡起一根树枝，在篝火边干燥的土地上画了起来。她画了一个简单的方框，然后在方框里画了几个小格子。

"来，我们一个个说。"黛琳用树枝指着第一个格子，"第一个是 buildConfig——这个最重要。你知道库模块为什么有时候不生成 BuildConfig 类吗？"

洛芙想了想："因为……库模块不是直接运行的？"

"对了一半。"黛琳点点头，"库模块会被打包进 app 模块，所以默认情况下，库模块不会生成自己的 BuildConfig 类，避免和 app 模块的 BuildConfig 冲突。但是如果你在库里确实需要用到版本号、构建类型这些信息，就可以手动打开这个开关。"

"那怎么开？"洛芙问。

希尔立刻把自己的笔记本递过来："看代码。"

```kotlin
// library 的 build.gradle.kts
android {
    library {
        buildFeatures {
            buildConfig = true
        }
    }
}
```

"就这么简单？"洛芙有点不敢相信。

"嗯，就两行。"希尔说，"但关键是——你得知道什么时候该开，什么时候不该开。我的经验是：如果你的库是对外发布的 SDK，一定要开；如果是纯工具库，只是被 app 引用，那就别开。"

伊莎在旁边补充道："而且开了之后，你就可以在库里写这样的代码——"

```kotlin
// 在库模块中读取版本信息
val versionName = BuildConfig.VERSION_NAME
val versionCode = BuildConfig.VERSION_CODE
val buildType = BuildConfig.BUILD_TYPE // "debug" 或 "release"
```

"哇，感觉和 app 模块一样了。"洛芙感叹。

"但还是有区别。"黛琳用树枝在地上敲了敲，"库模块的 BuildConfig 是在编译时生成的，和 app 模块的 BuildConfig 放在不同的命名空间里，所以不会冲突。"

接下来，黛琳指向她画的第二个格子："第二个是 enableR8——这个是控制 R8 混淆的。"

"R8 我知道！"洛芙这次有底气了，"是那个代码混淆器，把类名、方法名变成 a、b、c 那个对吧？"

"对，就是它。"黛琳说，"库模块默认是启用 R8 的，但有时候我们需要它保持可读——比如这是一个调试库，或者我们希望别人能看到原始的类名。"

"那关掉不就行了？"洛芙说。

"可以关，但有代价。"希尔接过话头，"R8 不仅做混淆，还做优化——比如删除未使用的代码、简化字符串。如果你关掉 R8，库文件体积会变大，加载速度也会变慢。"

黛琳点点头，在地上写下代码示例：

```kotlin
android {
    library {
        buildFeatures {
            buildConfig = true
            enableR8 = true  // 默认值，可以不写
        }
    }
}
```

"第三个是 resValue。"黛琳说，"这个很有趣——它允许库模块向 app 模块添加资源值。"

"资源值？"洛芙眨眨眼，"比如字符串？颜色？"

"对，比如字符串、尺寸、颜色。"伊莎说，"你想啊，你的库可能需要一些默认的字符串——报错信息啦、默认的提示文案啦——你就可以在库里直接定义，app 模块不需要再重复定义一遍。"

黛琳在地上写出示例：

```kotlin
android {
    library {
        buildFeatures {
            buildConfig = true
            enableR8 = true
            resValue = true
        }
        
        // 在库中定义资源值
        resValue("string", "lib_default_name", "MyLibrary")
        resValue("color", "lib_primary_color", "#FF6200EE")
        resValue("dimen", "lib_default_padding", "16dp")
    }
}
```

"这样 app 模块就能直接用这些资源？"洛芙问。

"对，直接用，不需要任何 import。"希尔说，"但是有个坑——如果 app 模块也定义了同名的资源，app 的会覆盖库的。这就是资源优先级的规则。"

洛芙若有所思："原来如此。那还有其他的 feature 吗？"

黛琳又在方框里画了一个格子："还有一个是 testConfig，不过这个比较少见，主要是配置测试相关的特性。"

"那我们是不是可以把这些 features 组合起来，根据不同的 flavor 开启不同的特性？"洛芙灵机一动。

黛琳露出赞许的微笑："对，这就是进阶用法。比如——"

```kotlin
android {
    library {
        // 为 debug flavor 开启 BuildConfig，方便调试
        defaultConfig {
            buildFeatures {
                buildConfig = true
            }
        }
        
        // release flavor 关闭 BuildConfig，减少体积
        release {
            buildFeatures {
                buildConfig = false
            }
        }
        
        // 同时为不同的 flavor 配置不同的资源值
        flavorDimensions += "environment"
        productFlavors {
            create("demo") {
                dimension = "environment"
                // demo 版添加特定的资源值
                resValue("string", "flavor_name", "Demo Version")
            }
            create("full") {
                dimension = "environment"
                // 完整版添加不同的资源值
                resValue("string", "flavor_name", "Full Version")
            }
        }
    }
}
```

"这样看来，LibraryBuildFeatures 和 LibraryBaseFlavor 配合起来，能做很多灵活的配置啊。"洛芙感叹道。

"它们两个确实是好搭档。"伊莎说，"Flavor 决定了你有哪些构建变体，Features 决定每个变体有什么特性。一个管『有没有』，一个管『开不开』。"

希尔不知道什么时候已经把电脑拿出来了："我给你们看一个实际的例子——这是我之前做的一个 SDK 项目的完整配置。"

```kotlin
android {
    namespace = "com.example.mylibrary"
    
    compileSdk = 34
    
    defaultConfig {
        minSdk = 21
        testInstrumentationRunner = "androidx.test.runner.AndroidJUnitRunner"
        
        // 库的基本配置
        consumerProguardFiles("consumer-rules.pro")
    }
    
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
    
    kotlinOptions {
        jvmTarget = "17"
    }
    
    // LibraryBuildFeatures 配置
    library {
        // 开启 BuildConfig，方便库内部调试
        buildFeatures {
            buildConfig = true
            // enableR8 = true // 默认值
            resValue = true
        }
        
        // 为不同的 flavor 定义不同的资源值
        resValue("string", "lib_version", "1.0.0")
        resValue("color", "lib_accent", "#FF4081")
        
        // 不同 flavor 的不同配置
        flavorDimensions += "version"
        productFlavors {
            create("free") {
                dimension = "version"
                // 免费版不生成 BuildConfig，减少体积
                buildFeatures {
                    buildConfig = false
                }
                resValue("string", "flavor_tag", "Free")
            }
            create("pro") {
                dimension = "version"
                // Pro 版开启 BuildConfig
                buildFeatures {
                    buildConfig = true
                }
                resValue("string", "flavor_tag", "Pro")
            }
        }
    }
}
```

"这个例子好全面。"洛芙看得眼睛发亮，"感觉把学到的东西都串起来了。"

"而且要注意一点——"黛琳重点强调，"这些 features 都是在编译时生效的，不是运行时。所以你没法在代码里动态判断某个 feature 开还是关，它们是构建配置的一部分。"

洛芙点点头，表示理解。她抬头看向天空，星星比刚才更亮了。

"对了，还有一件事。"希尔突然想起什么，"有时候你会遇到 buildFeatures 报错的情况——比如你在 library {} 块外面写 buildFeatures，那是不对的。features 必须写在 library {} 里面。"

"这是很多人容易犯的错误。"黛琳补充道，"因为 app 模块的 buildFeatures 是写在 defaultConfig 或 buildTypes 里面的，但 library 模块是写在 library {} 里面的。结构不一样。"

"我记得了！"洛芙认真地说。

伊莎轻轻拍了拍洛芙的肩膀："今天的内容很实用吧？LibraryBuildFeatures 虽然看起来简单，但用好了能让你的库模块更加灵活。"

夜空下，四个人又聊了一会儿别的，直到黛琳打了个哈欠："好了，今天就到这里吧。深夜了，星星很美，但也要休息了。"

洛芙最后望了一眼湖面——夜雾更浓了，篝火的光在雾中显得更加柔和。她在心里默默整理着今天学到的知识：buildConfig 控制 BuildConfig 生成，enableR8 控制 R8 混淆，resValue 允许添加资源值，而这些都可以和 flavor 配合使用，做出灵活的配置。

这就是库模块的构建魔法呀。

---

> 学习建议：LibraryBuildFeatures 是 Android 库模块配置的核心，理解它与 LibraryBaseFlavor 的配合使用是关键。实际项目中，建议为对外发布的 SDK 开启 buildConfig，为纯工具库关闭 buildConfig 以减少体积。resValue 虽然方便，但要注意与 app 模块的资源优先级问题。

## 今日关键词

**LibraryBuildFeatures** - Android 库模块中配置构建特性的 DSL 接口，控制库模块的 BuildConfig 生成、R8 混淆、资源值等构建特性。

**BuildConfig** - Android 构建系统自动生成的类，包含版本号、构建类型等信息。在库模块中需要显式开启 buildConfig = true 才会生成。

**R8** - Google 推出的代码混淆器和优化器，默认在库模块中启用，可通过 enableR8 = false 关闭（但不推荐）。

**resValue** - 允许库模块向 app 模块添加资源值（字符串、颜色、尺寸等）的配置开关。

**flavorDimensions** - 产品风味的维度定义，用于创建多个风味的组合。

**productFlavors** - 产品风味配置块，可在库模块中定义不同的风味变体。

**buildFeatures** - 构建特性配置块，用于开启或关闭各种构建特性。
