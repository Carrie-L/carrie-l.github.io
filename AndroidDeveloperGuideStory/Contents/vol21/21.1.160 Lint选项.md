# 21.1.160 Lint选项

太阳渐渐升高，薄雾彻底散开了。

洛芙端着一杯热可可，坐在折叠桌前笔记本电脑屏幕发出的光映在她脸上。刚才黛琳讲解了 Lint 是什么，但它具体能配置哪些选项、这些选项分别控制什么行为，她们还没有详细展开。

"黛琳，"洛芙托着腮帮子，"刚才你说的那个 lint { } 配置块，里面到底能放什么？我看刚才的示例里只有几行，但感觉应该有很多可以配的吧？"

黛琳点了点头，把笔记本转过来，指着屏幕上的一段代码：

"问得好。其实 LintOptions 有很多配置项，每个都能精细控制 Lint 的行为。我们来一个个看——先从最常用的几个开始。"

---

## abortOnError：构建的"紧急制动"

黛琳在电脑上敲了几下，调出一个配置示例：

```kotlin
android {
    lint {
        // 当发现错误时，是否中止构建
        abortOnError = true
    }
}
```

"看到这个 `abortOnError` 了吗？"黛琳解释道，"它的意思是——当 Lint 发现错误时，是否要停止构建。"

"紧急制动？"伊莎抬起头，"这个比喻很形象啊。"

"对的，"黛琳点点头，"你可以把它想象成露营时的火警警报。如果设置为 true，就像在你帐篷里装了报警器——一旦发现火星（代码错误），立刻让所有人停止手头的事（中止构建）。如果设置为 false，报警器只会响，但还是会让大家继续搭帐篷（继续构建）。"

洛芙眨了眨眼："那一般来说要设成 true 吗？"

"看项目阶段，"希尔接过话头，"开发期间通常设为 false，让你能先跑通流程再说。但在上线前的最终打包——比如打 release 包——一定要设为 true，任何错误都不能放过。"

"我试试看！"希尔在笔记本上敲了一段模拟代码，然后展示了运行结果：

```
./gradlew assembleRelease

> Task :app:lint
> Running lint...
> Issues found: 1 error, 3 warnings
> BUILD FAILED: abortOnError=true, stopping build

```

"看，这就是 abortOnError = true 的效果，"希尔说，"一旦有错误就直接失败。"

---

## warningsAsErrors：把警告当错误

"还有一个很常用的选项，"黛琳继续道，"叫 warningsAsErrors。"

她在屏幕上打出配置：

```kotlin
android {
    lint {
        // 将警告视为错误
        warningsAsErrors = true
    }
}
```

洛芙歪着头："这个又是干嘛的？"

"简单说，它会把警告也提升为错误，"黛琳解释道，"默认情况下，Lint 发现问题会分两种：Error（错误）和 Warning（警告）。Error 会导致构建失败，Warning 只会提示你，但不会阻止构建。"

"但有时候，"黛琳补充道，"我们希望更严格一点——哪怕是警告也不放过。这时候就把 warningsAsErrors 设为 true。"

伊莎轻轻敲了敲咖啡杯边缘："就像露营时的卫生标准？错误是明火，警告是乱丢的垃圾——平时不管它也没事，但如果要求严格起来，乱丢垃圾也不行。"

"伊莎的比喻太到位了！"希尔笑道，"没错，warningsAsErrors 就是那个'严格模式'。"

黛琳点点头："一般在 CI/CD 流水线或者 release 构建时，很多人会打开这个选项。"

---

## checkReleaseBuilds：Release构建要检查吗？

"接下来是这个，"黛琳滑动屏幕，"checkReleaseBuilds。"

```kotlin
android {
    lint {
        // 是否在 release 构建时运行 Lint
        checkReleaseBuilds = true
    }
}
```

洛芙举手："这个问题我知道！是不是说——要不要在打 release 包的时候跑 Lint？"

"对，就是这个意思，"黛琳给了她一个赞许的眼神，"默认是 true，意味着你打 release 包时会自动跑 Lint 检查。但有时候如果你想跳过检查（比如临时赶时间），可以设为 false。"

"不过我们不推荐在正式发布时关闭它，"希尔补充道，"release 版本尤其需要 Lint 把关，因为这是面向用户的最终产品。"

---

## disable 与 enable：开关特定检查

"接下来这个功能很实用，"黛琳微微一笑，"有时候我们觉得某个 Lint 检查不太适合自己项目，想把它关掉——这时候就用 disable。"

```kotlin
android {
    lint {
        // 禁用特定的 Lint 检查
        disable = ['TypographyFractions', 'RtlHardcoded', 'GoogleAppIndexingWarning']
    }
}
```

洛芙看着这一串字符串："这些是什么？"

"这些是 Lint 检查的 ID，"黛琳解释道，"每个检查都有一个唯一的名字。比如 TypographyFractions 是关于排版分数的，RtlHardcoded 是关于从右到左布局硬编码的，GoogleAppIndexingWarning 是关于 Google 应用索引的。"

"关掉它们？"洛芙有些疑惑，"这样不好吧？"

"不是随便关的，"黛琳认真地说，"当你确定某个检查在你的项目场景下不适用，或者你有更好的处理方式时，可以禁用它。但一定要有充分的理由。"

"那 enable 呢？"伊莎问。

"enable 正好相反——"黛琳调出另一个示例，"它是用来显式开启某些检查。默认是开启所有检查，但如果你想只跑特定的检查，可以用这个："

```kotlin
android {
    lint {
        // 只启用特定的 Lint 检查
        enable = ['TypographyFractions', 'RtlHardcoded']
    }
}
```

"这就像露营时的分工表，"伊莎说，"disable 是说'这个人今天不用值班'，enable 是说'今天只让这几个人值班'。"

---

## check：精细控制检查范围

"如果要更精细地控制，可以用 check，"黛琳调出最后一段配置：

```kotlin
android {
    lint {
        // 配置检查的严格程度
        check = ['TypographyFractions', 'RtlHardcoded']
    }
}
```

"这个和 enable 有什么区别？"洛芙问。

"好问题，"黛琳点点头，"enable 是'在默认基础上额外开启'，而 check 是'只检查这些'。如果你用 check = [...], 那就只跑列出的这些检查，其他的全部跳过。"

"这个在什么场景用？"希尔问。

"比如你正在修复某一类问题，想只跑相关的检查确认修复效果，"黛琳解释道，"或者某个模块正在重构，想只检查那个模块相关的规则。"

---

## 完整配置示例：像食谱一样组合

"好了，我们把刚才学的组合起来，"黛琳在白板上画了一个示意，"来看看一个完整的 LintOptions 配置长什么样："

```kotlin
android {
    lint {
        // 核心行为控制
        abortOnError = true
        warningsAsErrors = true
        checkReleaseBuilds = true
        
        // 检查项开关
        disable = ['TypographyFarnings', 'RtlHardcoded']
        enable = ['GoogleAppIndexingWarning']
        check = ['MissingTranslation', 'ExtraTranslation']
        
        // 输出控制
        textReport = true
        xmlReport = false
        htmlReport = true
    }
}
```

"等等，"洛芙注意到几个新东西，"textReport、xmlReport、htmlReport 是干嘛的？"

"啊，这些是输出格式控制，"希尔抢答道，"Lint 检查完后可以生成不同格式的报告——"

- textReport = true → 生成纯文本报告
- xmlReport = true → 生成 XML 报告（适合 CI 系统解析）
- htmlReport = true → 生成 HTML 报告（适合人类阅读）"

"默认通常都开一个，方便不同场景使用，"希尔补充道。

---

## 实际演练：配置自己的 Lint 规则

"光说不练假把式，"希尔活动了一下手指，"我们来实际配一个看看。"

她打开项目目录下的 build.gradle.kts 文件，加入了以下配置：

```kotlin
android {
    lint {
        abortOnError = false  // 开发期间暂时不中断
        warningsAsErrors = true  // 警告也要认真对待
        checkReleaseBuilds = true  // release 必须检查
        
        // 禁用一些不适用的检查
        disable += "TypographyFractions"
        disable += "RtlHardcoded"
        
        // 开启我们关注的检查
        enable += "GoogleAppIndexingWarning"
        
        // 生成多种格式的报告
        textReport = true
        htmlReport = true
    }
}
```

"运行一下看看效果？"希尔看向其他人。

黛琳点头。希尔在终端执行：

```
./gradlew lint

> Task :app:lint
> Running lint...
> Warning: MissingTranslation in src/main/res/values-zh/strings.xml
> Warning: HardcodedText in MainActivity.kt:45
> Error found, aborting: checkReleaseBuilds=true and release build
> BUILD FAILED

```

"哇，真的检测出问题了！"洛芙惊呼。

"这还是我们故意设置的，"希尔笑道，"warningsAsErrors = true 把警告也变成错误了。"

---

## 按buildType差异化配置

"还有一点很实用，"黛琳突然想起什么，"Lint 配置可以按 buildType 差异化。"

她在屏幕上打出另一个示例：

```kotlin
android {
    lint {
        abortOnError = true
        
        // debug 构建不中断，只警告
        checkDebugBuilds = false
        
        // release 构建必须严格
        checkReleaseBuilds = true
    }
}
```

"checkDebugBuilds？"洛芙问。

"这个控制 debug 构建是否检查 Lint，"黛琳解释道，"通常设为 false，因为开发期间频繁构建，不想每次都被 Lint 中断。release 构建再用严格模式。"

---

## 洛芙的动手时刻

"好了，"黛琳看向洛芙，"你来试着配一个？要求是：release 构建时 abortOnError = true，debug 时 false；把 'MissingTranslation' 检查禁用掉；生成文本报告。"

洛芙点点头，在笔记本上敲了起来：

```kotlin
android {
    lint {
        abortOnError = true  // release 必须严格
        checkReleaseBuilds = true
        
        // 禁用特定检查
        disable += "MissingTranslation"
        
        // 生成文本报告
        textReport = true
    }
}
```

"等等，你这个 abortOnError 是全局的，"希尔指出问题，"如果要区分 debug 和 release，需要在 buildTypes 里单独配。"

"啊，对！"洛芙吐了吐舌头，"忘了这个。"

她重新修改：

```kotlin
android {
    lint {
        checkReleaseBuilds = true
        disable += "MissingTranslation"
        textReport = true
    }
}

android.buildTypes {
    release {
        isMinifyEnabled = true
        lint {
            abortOnError = true
            warningsAsErrors = true
        }
    }
    debug {
        lint {
            abortOnError = false
            warningsAsErrors = false
        }
    }
}
```

"这样就对了！"黛琳露出赞许的笑容，"在 buildTypes 里面单独配置 lint，debug 和 release 各自有各自的规则。"

## 知识总结：Linting 的最佳实践

伊莎总结道："经过今天的练习，我们学到了 LintOptions 的核心配置项——"

| 配置项 | 作用 | 推荐值 |
|--------|------|--------|
| abortOnError | 发现错误时是否中止构建 | release: true, debug: false |
| warningsAsErrors | 是否把警告当错误 | CI/release: true |
| checkReleaseBuilds | release构建是否检查 | true |
| disable | 禁用特定检查 | 按项目需要 |
| enable | 启用特定检查 | 按项目需要 |
| check | 只检查指定项 | 调试特定问题时用 |
| textReport/htmlReport/xmlReport | 报告输出格式 | 按需开启 |

"最关键的思路是，"黛琳补充道，"不同阶段用不同配置——开发时宽松、发布时严格。"

---

> 学习建议：建议在真实项目中实践这些配置，从一个简单的 abortOnError = true 开始，逐步添加其他选项。关注项目在 CI 环境下的 Lint 报告，根据报告不断优化检查规则。

## 今日关键词

- **LintOptions** - Android Gradle DSL 中用于配置 Lint 静态代码分析工具的类，可精细控制检查行为
- **abortOnError** - LintOptions 属性，发现错误时是否中止构建
- **warningsAsErrors** - LintOptions 属性，是否将警告视为错误
- **checkReleaseBuilds** - LintOptions 属性，是否在 release 构建时运行 Lint
- **checkDebugBuilds** - LintOptions 属性，是否在 debug 构建时运行 Lint
- **disable** - LintOptions 方法，禁用指定的 Lint 检查项
- **enable** - LintOptions 方法，启用指定的 Lint 检查项
- **check** - LintOptions 方法，只运行指定的检查项（其他跳过）
- **textReport** - LintOptions 属性，控制是否生成纯文本格式的 Lint 报告
- **htmlReport** - LintOptions 属性，控制是否生成 HTML 格式的 Lint 报告
- **xmlReport** - LintOptions 属性，控制是否生成 XML 格式的 Lint 报告
- **buildType** - Gradle 构建类型（debug/release），可分别配置不同的 Lint 规则
- **静态代码分析** - 不执行程序，通过分析代码结构发现潜在问题的技术
