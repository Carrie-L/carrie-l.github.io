# 21.1.175 预制件

傍晚的湖面像打翻的调色盘，橙红、粉紫、淡金交织在一起。

洛芙双手撑在草地上，仰头看着天空从浅蓝慢慢染上晚霞的颜色。她吸了吸鼻子，空气里飘着烤鱼的香味——不远处有人开始准备晚餐了。

“原来下午不知不觉就过去了啊。”洛芙伸了个懒腰，肩膀发出轻微的咔嗒声。

黛琳把白板笔盖好，放回笔盒里：“那我们今天就到这里？先把东西收——”

“等一下！”希尔按住黛琳的手，“刚才说的 Packaging Options 我还没完全消化呢，你就又想跑？再讲一个，就一个！”

伊莎笑着把垂到眼前的发丝别到耳后：“希尔今天特别积极呢。”

“因为明天下午我要去参加一个 Android 开发 meetup，想多学点东西嘛。”希尔吐了吐舌头，“而且我们不是一直在聊构建配置吗？刚才讲的那个排除资源的功能好酷，再讲一个类似的嘛~”

黛琳看了希尔三秒钟，终于还是把白板笔拿了出来：“真拿你没办法。那我们今天最后一个话题——Prefab。”

“Prefab？”洛芙歪着头，“又是新的 DSL？是做什么的？”

“你在写代码的时候有没有遇到过这种情况，”黛琳在白板上写下 "Prefab" 三个字母，“你想用某个原生库，比如 Google 的 ML Kit ，或者是 Unity 打包出来的 AAR，但这些库里面包含了 C/C++ 的原生代码，你想在 Kotlin 或 Java 代码里调用它们，却不知道该怎么把这些原生代码链接进来。”

洛芙点点头：“啊，有的！我之前想用一个图像处理的库，明明 AAR 导入了，但 native 方法就是调用不了。”

“那就是 Prefab 要解决的问题。”黛琳敲了敲白板，“Prefab 是 Android Gradle Plugin 提供的 DSL，专门用来配置 AAR 中包含的原生库。它让你可以把原生代码像普通依赖一样导入进来，自动处理头文件、链接库等等。”

伊莎托着腮帮子：“听起来像是指挥官在调配资源呢。”

“比喻得不错。”黛琳笑了笑，“我们来看具体怎么用。首先，你的项目必须启用 native 构建：

```kotlin
android {
    // 启用 native 构建支持
    buildFeatures {
        prefab = true
    }
}
```

然后在你需要引用原生库的模块里，添加依赖并配置 Prefab 包名：

```kotlin
android {
    defaultConfig {
        // 配置支持的 CPU 架构
        ndk {
            abiFilters += listOf("armeabi-v7a", "arm64-v8a", "x86", "x86_64")
        }
    }
}

dependencies {
    // 引入包含原生代码的 AAR
    implementation("com.google.mlkit:pose-detection:18.0.0-beta3")
}
```

希尔敲了敲笔记本：“等等，这里面好像没看到 Prefab 的配置啊？”

“对，Prefab 有两种配置方式。”黛琳画了一幅图，“一种是自动模式——只要 AAR 本身包含了 Prefab 配置文件（就是 aar_metadata.jar），Gradle 会自动读取并暴露原生库给你。这种情况下你甚至不需要写任何 Prefab 相关的代码。”

```kotlin
// 自动模式：Gradle 自动发现并配置 Prefab 包
// 你只需要确保 buildFeatures { prefab = true } 启用即可
// 链接时自动使用 AAR 中的原生库
```

“另一种是手动模式，”黛琳继续说，“当 AAR 没有自带 Prefab 配置，或者你想覆盖默认行为时，可以用 Prefab DSL 手动配置：

```kotlin
android {
    buildFeatures {
        prefab = true
    }

    // 手动配置 Prefab 包
    prefab {
        // 包名（来自 AAR 的 prefab package name）
        create("my-custom-library") {
            // 指定 AAR 依赖
            libraryName = "mynative"  // CMake 或 ndk-build 生成的库名
        }
    }
}
```

洛芙盯着白板：“这个 `libraryName` 是做什么的？”

“每个 Prefab 包都有一个或多个原生库文件。`libraryName` 指定你要暴露的是哪个库。”黛琳解释道，“比如你用 ndk-build 编译出一个 libmynative.so，那 libraryName 就是 'mynative'。”

伊莎举起手：“那如果我想同时暴露多个库怎么办？”

“那就创建多个 `create()` 块：

```kotlin
prefab {
    create("lib-one") {
        libraryName = "mynative1"
    }
    create("lib-two") {
        libraryName = "mynative2"
    }
}
```

希尔眼睛亮了起来：“我明白了！这样就可以把一个 AAR 里的多个原生库分别暴露出去。那——这些原生库到底怎么在代码里用啊？”

“好问题。”黛琳把白板擦干净，画了一幅新的图，“ Prefab 的核心工作流程是这样的：

```mermaid
flowchart LR
    A["build.gradle\n声明 Prefab 依赖"] --> B["AGP 读取 AAR 中\n的 Prefab 配置"]
    B --> C["自动生成 CMake\n或 ndk-build 导入脚本"]
    C --> D["native 代码通过\n#include 引用头文件"]
    D --> E["链接器自动\n链接原生库"]
```

（图 1 对应上述代码配置）

“也就是说，”洛芙好像抓住了什么重点，“我只需要在 build.gradle 写几行配置，然后就可以在 C++ 代码里直接 `#include <mlkit/pose_detection.h>` 这样引用？”

“对的。这就是 Prefab 的精髓——把复杂的 native 集成变成像普通依赖一样简单。”

湖面上晚霞的颜色更深了，几只夜鹭从芦苇丛中飞出，发出沙哑的叫声。远处的露营地上亮起了几点灯光，是开始准备晚饭的营火。

“那——如果我想检查某个 AAR 支不支持 Prefab，有没有办法？”希尔问。

“有两种方法。”黛琳伸出两根手指，“第一，看 AAR 里有没有 `prefab PrefabConfig.cmake` 这个文件（在 aar-metadata.jar 里）。第二，用 `./gradlew app:dependencies` 查看依赖树，如果看到 `prefab` 标记，说明这个依赖支持 Prefab。”

“如果不支持呢？”洛芙问。

“那就需要手动配置 CMake 或 ndk-build，把原生库文件拖进去自己链接。或者——更推荐的做法是，找一个支持 Prefab 的替代库。”

希尔打开笔记本，开始敲代码：“让我整理一下今天学到的——”

```kotlin
// 基础配置：启用 Prefab
android.buildFeatures.prefab = true

// 自动模式（常用）
// 只需要添加 AAR 依赖，Gradle 自动处理
dependencies {
    implementation("com.google.mlkit:pose-detection:18.0.0-beta3")
}

// 手动模式（当 AAR 不支持自动配置时）
android.prefab {
    create("mylib") {
        libraryName = "native-lib"
    }
}
```

伊莎突然说：“我有个问题——如果我用的是老版本的 AAR，里面没有 Prefab 配置，但我又不想换新版本，有什么办法让它支持 Prefab 吗？”

黛琳想了想：“可以自己写一个 Prefab 配置文件。Prefab 的配置文件是标准的 CMake module，你只需要创建一个 `PrefabConfig.cmake` 文件，放到你的模块目录下，然后在 build.gradle 里引用：

```kotlin
prefab {
    create("my-patched-lib") {
        // 指向你手动写的 Prefab 配置
        cmakeFlags += "-DCMAKE_MODULE_PATH=${projectDir}/prefab"
    }
}
```

不过这属于高级用法，一般不推荐。”

洛芙举起手：“我我我——还有个问题！那 Gradle 是怎么知道一个 AAR 里面有哪些原生库的呢？”

“这就是 Prefab 的核心机制了。”黛琳在白板上画了一个新的结构图，“每个支持 Prefab 的 AAR 都会在它的 aar-metadata.jar 里放一个 `PrefabConfig.cmake` 文件，就像这样：

```mermaid
classDiagram
    class AAR {
        +Java/Kotlin 代码
        +原生库 .so 文件
        +aar-metadata.jar
    }
    class "aar-metadata.jar" {
        +PrefabConfig.cmake
        +PrefabPackageConfig.cmake
    }
    class "PrefabConfig.cmake" {
        +add_library() 定义
        +target_link_libraries() 定义
    }
```

（图 2 对应 AAR 内部 Prefab 配置结构）

“Gradle 会读取这个配置文件，然后自动生成给你用的 CMake 导入脚本。”

希尔补充道：“所以那些 ML Kit 的库才能开箱即用对吧？因为 Google 已经在 AAR 里打包好了这些配置文件。”

“没错。”黛琳点头，“这就是为什么我建议你们优先使用 Google 官方的库——它们都支持 Prefab，省了很多配置功夫。”

洛芙总结道：“所以 Prefab 就是让原生库能像普通依赖一样被自动发现和链接的功能，对吧？”

“没错。”黛琳把白板笔收好，“它是 Android Gradle Plugin 用来统一管理 native 依赖的接口。不管是 Google 的 ML Kit、Unity，还是你们公司内部的原生库，只要支持 Prefab，就可以用相同的方式集成。”

希尔伸了个大大的懒腰：“今天又学到新东西了！感觉 Native 开发也没那么可怕了。”

“走吧，去吃饭。”伊莎站起来，拍了拍裙子上的草屑，“今天好像有烤鱼，我闻到了！”

晚霞把四个女孩的影子拉得很长。她们收拾好白板和笔记本，沿着湖边小径走向露营地的公共炊事房。湖水在晚风中轻轻荡漾，倒映着已经变成深蓝色的天空。

## 今日关键词

- **Prefab**：Android Gradle Plugin 提供的 DSL，用于配置 AAR 中包含的原生库（C/C++ 代码），支持自动发现和链接原生依赖
- **buildFeatures.prefab**：启用 Prefab 功能的开关，需在 build.gradle 中设置为 `true`
- **ndk.abiFilters**：配置支持的 CPU 架构（armeabi-v7a、arm64-v8a、x86、x86_64 等）
- **PrefabConfig.cmake**：Prefab 的配置文件，标准 CMake module 格式，定义原生库的导入规则
- **AAR (Android Archive)**：Android 库的打包格式，可包含 Java/Kotlin 代码和原生代码
- **CMake**：C/C++ 的构建系统，常用于 Android 原生开发
- **ndk-build**：Android NDK 的另一种构建系统，与 CMake 并行使用
- **libraryName**：Prefb DSL 中指定要暴露的原生库名称
- **native library**：原生库，即编译好的 .so 文件
- **header file**：头文件，.h 文件，用于 C/C++ 代码的声明
- **abi (Application Binary Interface)**：应用二进制接口，不同 CPU 架构对应的二进制格式
- **aar-metadata.jar**：AAR 中的元数据容器，包含 Prefab 配置文件
- **add_library()**：CMake 命令，用于声明要链接的原生库
- **target_link_libraries()**：CMake 命令，用于指定库的依赖关系
- **.so 文件**：Linux/Unix 动态库文件，Android 原生库的标准格式
