“版本还有讲究啊？”希尔不知道什么时候从帐篷里钻了出来，手里拿着一瓶冰镇可乐，瓶身上凝结着晶莹的水珠。

## 版本管理是什么？

黛琳把电脑放在一块平整的大石头上，屏幕在阳光下微微发亮。她打开了一个空白的Kotlin文件。

“在讲代码之前，我先问你们一个问题。”黛琳双手交叉抱在胸前，“你们有没有遇到过这种情况？同一个App，在不同版本的Android Gradle Plugin上，能跑出完全不一样的结果？”

洛芙举手：“我！我之前做一个项目，升级了AGP版本之后，编译突然就报错了！”

“这就是版本差异造成的问题。”黛琳点点头，“Android Gradle Plugin——简称AGP——会不断更新，有时候会添加新API，有时候会废弃旧API。如果我们写一个插件，想要同时支持多个AGP版本，就必须知道当前项目用的是哪个版本。”

她俯下身，在键盘上敲了几下：

```kotlin
// 获取当前AGP版本
val currentVersion = AndroidPluginVersion.getCurrent()
println("当前AGP版本: ${currentVersion}")
```

“你们看，这行代码就能获取当前项目的AGP版本。”黛琳直起身，让开屏幕让大家都能看到，“不过光知道版本号还不够——我们还得学会比较版本。”

## 版本号的秘密

伊莎凑近屏幕，她的猫耳朵随着动作轻轻抖了抖：“黛琳，这个版本号好像和我们平时说的不太一样？”

“对，这就是第一个要知道的知识点。”黛琳指着屏幕上的代码，“AGP的版本号是分三部分的：主版本号（major）、次版本号（minor）、补丁版本号（micro）。就像7.0.1，7是主版本，0是次版本，1是补丁版本。”

她重新敲了一段代码：

```kotlin
// 创建版本对象
val version7_0 = AndroidPluginVersion(7, 0)      // 7.0.0
val version7_0_1 = AndroidPluginVersion(7, 0, 1) // 7.0.1

// 这两个是等价的！
println(version7_0 == version7_0_1)  // 输出: true
```

洛芙眼睛一亮：“哦！7.0和7.0.0是一样的！”

“对，这就是内部表示的标准化。”黛琳说，“不管你写(7, 0)还是(7, 0, 0)，内部都会统一处理。所以(7, 0) == (7, 0, 0)是true。”

希尔可乐也不喝了，饶有兴趣地问：“那如果我们想用一些新版API，但又要兼容旧版本怎么办？”

“问得好。”黛琳打了个响指，“这就涉及到版本比较了。”

## 版本比较的艺术

黛琳在屏幕上画了一条看不见的线：“想象一下，我们在写一个插件，这个插件依赖某个特定版本的AGP才能工作。那我们就需要检查——当前AGP版本够不够新？”

```kotlin
// 假设我们需要一个最低AGP 7.0版本
val requiredVersion = AndroidPluginVersion(7, 0)
val currentVersion = AndroidPluginVersion.getCurrent()

// 版本比较 - 实现Comparable接口
if (currentVersion >= requiredVersion) {
    println("版本满足要求，可以继续！")
} else {
    println("版本太旧了，需要升级！")
}
```

洛芙举手：“这个>=是怎么比较的？”

“和数字比较一样。”黛琳解释道，“先比主版本号，主版本相同的话比次版本号，还相同的话比补丁版本号。这是自然排序——7.1比7.0大，8.0比7.0大。”

她停顿了一下，补充道：“不过，还有一个小陷阱很多人会踩到。”

“什么陷阱？”洛芙好奇地问。

“最高版本检查。”黛琳的表情变得认真起来，“假设你想让代码只在8.7系列版本运行，很多人会写：”

```kotlin
// ❌ 错误示例 - 这样写会漏掉8.7.1、8.7.2等补丁版本！
if (currentVersion.major == 8 && currentVersion.minor == 7) {
    // 只在8.7运行
}
```

“为什么错了？”希尔问。

“因为8.7.1也会被认为是8.7，但它其实是补丁版本。”黛琳摇头，“正确的做法是用一个稍高一点的版本来做exclusive比较：”

```kotlin
// ✅ 正确示例 - 使用exclusive bound
if (currentVersion >= AndroidPluginVersion(8, 7) && 
    currentVersion < AndroidPluginVersion(8, 8).alpha(1)) {
    // 只在8.7.x系列运行（包括8.7.0、8.7.1等）
}
```

洛芙若有所思：“8.8.alpha(1)又是什么？”

“这就是我们接下来要讲的——预览版本。”伊莎接过话头，她翻开素描本，上面画着一排不同颜色的小瓶子，“让我来解释这个吧！”

## 预览版本的彩色世界

伊莎的素描本上，不同颜色的瓶子整齐地排列着。

“想象一下，AGP的版本就像露营的不同阶段。”伊莎用手指着瓶子说，“最左边的是稳定版——就像我们已经搭好的帐篷，稳稳的，不会变。”

她指向中间的几个瓶子：“然后是alpha版——这是最早的测试版，就像我们在搭帐篷之前画的草图，可能会变很多很多次。”

“beta版呢？”洛芙问。

“beta版就像是草图确定之后，开始真正动手搭的阶段。”伊莎笑道，“这时候结构基本定了，但可能还会调整一下细节。”

“rc版就是release candidate——发布候选版。”希尔补充道，“基本上就是快要正式发布的状态了。”

黛琳点点头，接回话题：“在代码中，这些预览版本是通过特殊方法创建的。”

```kotlin
// 创建预览版本
val alpha5 = AndroidPluginVersion(7, 0).alpha(5)   // 7.0.0-alpha05
val beta2 = AndroidPluginVersion(7, 0).beta(2)    // 7.0.0-beta02
val rc1 = AndroidPluginVersion(7, 0).rc(1)        // 7.0.0-rc01
val dev = AndroidPluginVersion(7, 0).dev()        // 7.0.0-dev
```

“这些方法会返回一个带有预览类型的版本对象。”黛琳解释道，“你们可以通过preview和previewType属性来查看：”

```kotlin
val version = AndroidPluginVersion(7, 0).alpha(5)
println(version.major)        // 7
println(version.minor)        // 0
println(version.micro)        // 0
println(version.preview)      // 5
println(version.previewType)  // "alpha"
```

洛芙新奇地看着这些代码：“那这些预览版本怎么比较呢？”

“预览版本的比较规则是：稳定版 > rc版 > beta版 > alpha版 > dev版。”黛琳说，“也就是说，7.0.0比7.0.0-rc01大，7.0.0-rc01比7.0.0-beta02大。”

她在屏幕上画了一个简单的比较图：

```kotlin
// 版本层级（从高到低）
// 7.0.0 (stable) > 7.0.0-rc01 > 7.0.0-beta02 > 7.0.0-alpha05 > 7.0.0-dev

val stable = AndroidPluginVersion(7, 0)
val rc = AndroidPluginVersion(7, 0).rc(1)
val beta = AndroidPluginVersion(7, 0).beta(2)
val alpha = AndroidPluginVersion(7, 0).alpha(5)
val dev = AndroidPluginVersion(7, 0).dev()

println(stable > rc)    // true
println(rc > beta)     // true
println(beta > alpha)  // true
println(alpha > dev)   // true
```

## 实际应用：条件化功能

希尔急性子，看到这里就坐不住了：“好了好了，理论我懂了。那这到底能用来干嘛？”

黛琳笑着按下她的肩膀：“别急，马上就讲实际应用。”

她重新整理了一下屏幕上的代码：“想象你写了一个插件，这个插件依赖AGP 8.0新增的一个API。那你就需要在代码里判断——如果AGP版本低于8.0，就不能调用这个新API，要用旧方法代替。”

```kotlin
// 根据AGP版本使用不同的API
androidComponents.onEach {
    if (it.pluginVersion >= AndroidPluginVersion(8, 0)) {
        // 使用AGP 8.0+的新API
        it.onVariantProperties {
            // 新版API的逻辑
        }
    } else {
        // 使用兼容旧版本的逻辑
        it.onEachVariant {
            // 旧版API的逻辑
        }
    }
}
```

洛芙眼睛亮了：“这样就能让同一个插件支持多个AGP版本！”

“对，这就是版本检测的真正用途。”黛琳点头，“不过还有一点要记住——一旦某个API在稳定版中发布了，就要尽快去掉对预览版本的支持。”

“为什么？”洛芙不解。

“因为预览版不稳定，今天用起来没问题，明天可能就变了。”黛琳认真地说，“官方文档的建议是：当某个API在稳定版中可用后，就要把预览版的条件判断去掉，只用稳定版。”

```kotlin
// ❌ 旧代码 - 还在支持预览版
if (androidComponents.pluginVersion >= AndroidPluginVersion(7, 0).alpha(5)) {
    // 使用新API
}

// ✅ 新代码 - 只支持稳定版
if (androidComponents.pluginVersion >= AndroidPluginVersion(7, 0)) {
    // 使用新API
}
```

## 版本字符串

```kotlin
val version = AndroidPluginVersion(7, 0).alpha(5)

// 转换为字符串
println(version.version)  // "7.0.0-alpha05"

// 也能直接打印
println(version.toString())  // "AndroidPluginVersion(7, 0, 0, preview=5, previewType=alpha)"
```

“Maven格式？”洛芙歪着头。

“就是像7.0.0、7.0.1、8.0.0-alpha05这样的字符串格式，和Google Maven仓库里发布的版本号完全一致。”黛琳解释，“这样你就可以方便地存到配置文件里，或者显示给用户看了。”

## 完整示例：版本检测插件

希尔跃跃欲试：“能不能给我们看一个完整的例子？”

黛琳想了想：“好，我来写一个完整的示例，展示如何在实际插件中检测版本。”

她新建了一个文件，开始敲代码：

```kotlin
// 示例：开发一个支持多版本的插件
class MyPlugin {
    fun apply(project: Project) {
        val androidExtension = project.extensions.getByType(AndroidComponentsExtension::class.java)
        
        androidExtension.onEach {
            val version = it.pluginVersion
            
            // 获取版本信息
            val major = version.major
            val minor = version.minor
            val micro = version.micro
            val preview = version.preview
            val previewType = version.previewType
            
            println("检测到AGP版本: ${version.version}")
            
            // 根据版本做不同处理
            when {
                // 8.0+ 使用新版API
                version >= AndroidPluginVersion(8, 0) -> {
                    println("→ 使用AGP 8.0+新特性")
                }
                // 7.0-7.9 使用过渡API
                version >= AndroidPluginVersion(7, 0) -> {
                    println("→ 使用AGP 7.x兼容方案")
                }
                // 低于7.0给出警告
                else -> {
                    println("⚠️ 建议升级到AGP 7.0或更高版本")
                }
            }
        }
    }
}
```

“这段代码展示了插件开发中最常用的版本检测模式。”黛琳指着屏幕说，“先获取pluginVersion，然后根据版本高低做不同的处理——这就是条件化功能的核心。”

洛芙，认真看着代码：“感觉有点像……根据季节选衣服？”

“对！就是这种感觉。”伊莎笑着说，“AGP版本就像季节——不同版本要用不同的‘穿法’，这样才能既安全又高效。”

## 常见误区

太阳慢慢偏西，篝火堆旁边的温度却一点没降。黛琳收拾了一下笔记本电脑，准备做最后的总结。

“在使用AndroidPluginVersion时，有几个常见的误区要提醒你们注意。”

她伸出两根手指：

“第一，也是最常见的——不要直接比较版本字符串。”

```kotlin
// ❌ 错误 - 字符串比较不准确
if (currentVersion.toString() == "7.0.0") { }

// ✅ 正确 - 使用版本对象比较
if (currentVersion == AndroidPluginVersion(7, 0)) { }
```

“第二，注意previewType可能为null。”

```kotlin
// ❌ 错误 - previewType可能是null
if (version.previewType == "alpha") { }

// ✅ 正确 - 先检查是否为null
if (version.previewType != null && version.previewType == "alpha") { }
```

“第三，记得处理版本获取失败的情况。”

```kotlin
// 建议: 总是检查版本是否可用
try {
    val version = AndroidPluginVersion.getCurrent()
    // 使用版本
} catch (e: Exception) {
    println("无法获取AGP版本: ${e.message}")
}
```

## 今日关键词

- **AndroidPluginVersion**：Android Gradle Plugin版本的对象表示，用于在插件中检测和比较AGP版本
- **major（主版本号）**：版本号的第一位，表示重大变更（如7→8）
- **minor（次版本号）**：版本号的第二位，表示新功能（如7.0→7.1）
- **micro（补丁版本号）**：版本号的第三位，用于bug修复（如7.0.1）
- **preview（预览版本号）**：预览版的序号（如alpha05中的5）
- **previewType（预览类型）**：预览版的类型：alpha、beta、rc、dev或null
- **alpha()**：创建alpha预览版本的方法
- **beta()**：创建beta预览版本的方法
- **rc()**：创建release candidate预览版本的方法
- **dev()**：创建开发版本的方法
- **version（属性）**：返回Maven格式的版本字符串
- **getCurrent()**：companion object方法，获取当前AGP版本
- **条件化功能**：根据版本不同使用不同API实现的技术
- **优雅降级**：在新API不可用时使用旧方案兜底的策略
