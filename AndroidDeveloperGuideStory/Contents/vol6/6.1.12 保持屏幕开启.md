# 6.1.12 秋日午后的屏幕守护术

午餐的香气还萦绕在鼻尖，黛琳便携式炉具上的炖锅已经见底了。伊莎把最后一块番茄汁沾着面包吃完，满足地叹了口气。

“哎呀，中午的阳光太舒服了。”伊莎伸了个懒腰，直接躺在草地上，“要是能一直这样躺着就好了。”

希尔正在收拾餐具，她看了一眼手机：“你们看，这个食谱App又自动锁屏了。我刚才照着它做午餐的时候也是，老是得重新点亮屏幕，好麻烦啊。”

洛芙正在用树枝在地上画着圈圈，听到这话抬起头来：“希尔姐姐，既然你之前说FLAG_KEEP_SCREEN_ON可以让屏幕一直亮着，为什么不干脆让这个App一直保持屏幕亮着呢？”

黛琳正在把锅里的残羹倒进垃圾袋，听到这个问题，她的手顿了一下，随即转身走了过来。

“你这个问题问得很好，洛芙。”黛琳在洛芙身边坐下，阳光穿过稀疏的松枝，在她脸上投下斑驳的光影，“不过，让屏幕一直亮着，可不是随随便便就能做的事情哦。”

## 6.1.12.1 屏幕保持的基本原理

黛琳从背包里掏出她的笔记本电脑，放在膝盖上。洛芙凑过去看，屏幕上显示着Android开发的文档。

“在Android系统里，让屏幕保持开启，其实有两种不同的方法。”黛琳指着屏幕说道，“一种是上一章我们讲过的WakeLock，而另一种，就是我们今天要聊的——FLAG_KEEP_SCREEN_ON。”

伊莎也从草地上爬起来，好奇地凑了过来。

“这两者有什么区别吗？”伊莎问道。

“区别可大了。”黛琳微微一笑，“想象一下，你在一间教室里。如果是WakeLock，那就像是让整个教学楼的电力系统一直运转——哪怕教室里空无一人，所有的灯和空调都不能关。而FLAG_KEEP_SCREEN_ON呢，就像是在教室里贴了个'请保持开灯'的纸条，只影响你当前所在的这个房间。”

洛芙若有所思地点点头：“也就是说，FLAG_KEEP_SCREEN_ON只影响当前这个Activity，而WakeLock影响的是整个系统？”

“完全正确。”黛琳赞许地看了洛芙一眼，“FLAG_KEEP_SCREEN_ON是Window级别的，它只对设置了该flag的窗口生效。而WakeLock是PowerManager级别的，它会直接影响整个设备的电源状态。”

```kotlin
// 在Activity中使用FLAG_KEEP_SCREEN_ON
class MainActivity : AppCompatActivity() {
    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        
        // 方法一：在代码中直接设置
        window.addFlags(WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON)
    }
    
    override fun onDestroy() {
        super.onDestroy()
        // 注意：FLAG_KEEP_SCREEN_ON不需要手动清除
        // 当Activity销毁时，系统会自动移除这个flag
    }
}
```

“看这段代码，”黛琳指着屏幕上的Kotlin代码说道，“只需要调用`window.addFlags()`方法，传入`FLAG_KEEP_SCREEN_ON`常量，屏幕就会一直保持亮着。而且最方便的是，你不需要在Activity销毁时手动清除这个flag，系统会自动帮你处理。”

## 6.1.12.2 适用场景与注意事项

伊莎歪着头想了想：“那这个flag适合在什么情况下使用呢？”

“问得好。”黛琳点点头，“FLAG_KEEP_SCREEN_ON最适合的场景，是用户需要长时间盯着屏幕的时候。比如——”

“比如看食谱！”希尔抢着说道，“我做饭的时候老是要看步骤，屏幕一直暗掉真的很烦。”

“没错，还有看电子书、播放视频、显示导航信息，等等。”黛琳补充道，“这些场景都有一个共同点：用户正在主动使用设备，屏幕熄灭会打断用户体验。”

洛芙举手提问：“那有什么情况下不适合用吗？”

“有，而且很重要。”黛琳的表情变得认真起来，“如果你在后台服务中试图用FLAG_KEEP_SCREEN_ON来保持屏幕亮着，那可就大错特错了。”

“为什么？”洛芙不解地问。

“因为FLAG_KEEP_SCREEN_ON是Window级别的，它只对你设置的那个窗口有效。”黛琳耐心解释道，“如果你的Activity已经不可见了，这个flag就失效了。这时候如果你真的需要保持设备唤醒，就得用WakeLock——也就是我们上一章讲的内容。”

洛芙恍然大悟：“所以它们是互补的？FLAG_KEEP_SCREEN_ON适合前台使用，WakeLock适合后台服务？”

“完全正确！”黛琳打了个响指，“这就是Android设计这两个API的智慧所在——让开发者根据具体场景选择最合适的方案。”

## 6.1.12.3 电池影响与最佳实践

一直安静地听着的伊莎突然开口了：“我一直有个疑问——屏幕一直亮着，会不会很耗电啊？”

“这正是我想说的第三点。”黛琳点点头，表情变得有些严肃，“屏幕是手机最耗电的部件之一。长时间保持屏幕亮着，会大幅缩短电池续航。”

她打开电脑上的电池统计图表，指给三人看：“你们看，这是Android系统的电量消耗分布。屏幕通常会占到30%到50%的电量消耗。如果你的应用让屏幕一直亮着，用户的使用体验确实好了，但代价是他们的手机可能撑不到晚上就要充电。”

“那怎么办？”希尔皱起眉头，“难道就不能让屏幕一直亮着吗？”

“可以，但要有节制。”黛琳说道，“这里有几个最佳实践——”

**第一，能不用就不用。** 想想你的应用场景真的需要屏幕一直亮着吗？如果只是偶尔需要，可以考虑用WakeLock代替。

**第二，及时释放。** 如果你必须在某个Activity中保持屏幕亮着，至少要在不需要的时候尽快移除这个flag。

**第三，给用户选择。** 最好在你的应用设置中添加一个选项，让用户自己决定是否需要屏幕保持常亮。

```kotlin
// 更好的实践：在XML布局中使用android:keepScreenOn属性
<!-- activity_main.xml -->
<LinearLayout 
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    android:keepScreenOn="true">
    
    <!-- 内容区域 -->
    <TextView
        android:layout_width="wrap_content"
        android:layout_height="wrap_content"
        android:text="@string/cooking_steps" />
        
</LinearLayout>
```

“看这种方法，”黛琳指着代码说道，“在XML布局文件中直接使用`android:keepScreenOn`属性，效果和代码中设置flag是一样的，但语义更清晰——开发者一看就知道这个布局需要保持屏幕亮着。”

伊莎好奇地问：“这两种方法有区别吗？”

“本质上是一样的，”黛琳解释道，“都是在View的层级上设置一个flag。但XML的方式更直观，适合在布局层面就需要屏幕保持的场景。代码方式则更灵活，可以在运行时根据条件动态设置。”

## 6.1.12.4 反模式与重构对比

洛芙突然想到一个问题：“黛琳姐姐，我之前见过有人这么写——”

她拿起树枝，在地面上画了一行代码：

```
// 反模式：不推荐的做法
Handler().postDelayed({
    window.addFlags(WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON)
}, 5000)
```

“有人在Activity的onCreate里延迟5秒后才设置FLAG_KEEP_SCREEN_ON，这是为什么啊？”洛芙问道。

黛琳看完之后，摇了摇头：“这是一个比较奇怪的反模式。正常情况下，应该在onCreate中立即设置，而不是延迟。延迟设置不仅没有好处，反而可能导致用户体验不一致——用户可能在前5秒内就因为屏幕熄灭而离开你的应用了。”

“那正确的做法应该是什么呢？”希尔问道。

```kotlin
// 正确的做法
class RecipeActivity : AppCompatActivity() {
    
    private lateinit var binding: ActivityRecipeBinding
    
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityRecipeBinding.inflate(layoutInflater)
        setContentView(binding.root)
        
        // 在需要的地方立即设置，不需要延迟
        binding.recipeContent.keepScreenOn = true
    }
}
```

“还有另一个常见的反模式，”黛琳继续说道，“有人会在BroadcastReceiver中尝试设置FLAG_KEEP_SCREEN_ON——”

“这肯定不对吧？”伊莎猜测道。

“确实不对。”黛琳确认道，“BroadcastReceiver没有Window，你无法在Receiver中设置FLAG_KEEP_SCREEN_ON。如果你需要在后台保持屏幕亮着，必须使用WakeLock。”

洛芙总结道：“所以，FLAG_KEEP_SCREEN_ON只适用于有Window的场景，比如Activity、Dialog等？”

“没错，”黛琳欣慰地笑了，“你理解得很快。简单来说——前台用户交互用FLAG_KEEP_SCREEN_ON，后台服务用WakeLock。这样就不会混淆了。”

## 6.1.12.5 场景对比与选择指南

四人又聊了一会儿，黛琳觉得应该给她们一个更直观的对比总结。她打开笔记本电脑，画了一个表格：

| 特性 | FLAG_KEEP_SCREEN_ON | WakeLock |
|------|---------------------|-----------|
| 作用范围 | 仅当前Window | 整个系统 |
| 适用场景 | 用户主动使用设备 | 后台服务需要 |
| 电池影响 | 较高（屏幕常亮） | 可高可低（取决于类型） |
| 手动释放 | 不需要 | 必须 |
| 权限需求 | 不需要 | 需要`WAKE_LOCK`权限 |

“你们看，”黛琳指着表格说道，“最核心的区别就是作用范围。FLAG_KEEP_SCREEN_ON只影响设置了它的那个窗口——当用户离开这个窗口，或者关闭了这个Activity，屏幕保持自动失效。而WakeLock是系统级别的，除非你手动释放，否则它会一直生效。”

“还有一点很重要，”希尔补充道，“FLAG_KEEP_SCREEN_ON不需要任何权限，直接就可以用。而WakeLock需要申请`android.permission.WAKE_LOCK`权限，还需要在Manifest中声明。”

洛芙点点头：“我明白了！就像上次做饭的场景，如果是我在使用手机查看食谱，那就用FLAG_KEEP_SCREEN_ON；如果是后台有个下载服务需要保持CPU运行，那就用WakeLock。”

“完全正确！”黛琳赞许地竖起大拇指，“这就是Android设计的精妙之处——两个API各司其职，让开发者可以根据具体场景做出最佳选择。”

## 6.1.12.6 专业技术总结

本章我们学习了Android中保持屏幕开启的第一种方法——**FLAG_KEEP_SCREEN_ON**。

**核心要点：**

1. **FLAG_KEEP_SCREEN_ON是Window级别的flag**，只对设置它的Activity或Dialog生效
2. **不需要手动释放**，当Activity销毁时系统自动移除
3. **不需要任何权限**，直接可以使用
4. **最适合用户主动使用设备的场景**，如查看食谱、看视频、导航等
5. **不适合后台服务**，后台场景应使用WakeLock
6. **电池消耗较大**，应在不需要时尽快移除
7. **可以在代码或XML中设置**，推荐使用`android:keepScreenOn`属性

**与WakeLock的关系：**

- FLAG_KEEP_SCREEN_ON → 前台窗口级 → 用户可见
- WakeLock → 后台系统级 → 后台服务
