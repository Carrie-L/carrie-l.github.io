# 7.1.12 午餐时的超时提醒

中午的阳光正好，露营编程旅团的姑娘们开始准备午餐。黛琳便携式炉具上煮着意面，香味飘得老远。

“前台服务还有个很重要的概念——超时，”黛琳一边搅动着锅里的意面一边说道。

“超时？”洛FR抬起头，“服务还会超时？”

“会的，”黛琳点点头，“而且超时的后果还挺严重的。”

## 7.1.12.1 超时是什么

“在Android 12之前，前台服务可以一直运行，”黛琳说道，“但从Android 12开始，系统会给前台服务设置一个超时。”

“多长时间？”伊莎问。

“一般是20秒到几分钟，”黛琳说道，“取决于服务类型和系统状态。”

“为什么会超时？”洛FR问。

“因为系统怀疑你的服务可能出了问题，”黛琳解释道，“比如你启动了前台服务，但5分钟内既没有创建通知，也没有绑定任何东西，系统就会认为这是个'僵尸服务'，然后把它杀掉。”

## 7.1.12.2 超时回调

“如果超时了，系统会调用一个回调方法，”黛琳说道。

```kotlin
class MusicService : Service() {
    
    override fun onCreate() {
        super.onCreate()
        
        // Android 12+ 可以设置超时回调
        if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
            // 设置超时时间（可选）
            // 默认是几分钟，系统会根据情况调整
        }
    }
    
    // 超时回调
    override fun onTimeout(startId: Int, pendingIntent: PendingIntent) {
        // 这个方法会在超时后被调用
        
        // 你可以：
        // 1. 创建通知让它继续运行
        val notification = createNotification()
        startForeground(NOTIFICATION_ID, notification)
        
        // 或者：
        // 2. 直接停止服务
        stopForeground(STOP_FOREGROUND_REMOVE)
        stopSelf(startId)
    }
}
```

“收到这个回调后，你可以选择创建通知让服务继续运行，”黛琳解释，“或者直接停止服务。”

## 7.1.12.3 避免超时

“那怎么避免超时呢？”洛FR问。

“很简单：尽快调用startForeground()，”黛琳强调道。

```kotlin
override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
    // 立即调用startForeground！
    // 越早越好，绝对不能延迟！
    
    val notification = createNotification()
    startForeground(NOTIFICATION_ID, notification)
    
    // 然后再做其他工作
    doWork()
    
    return START_STICKY
}
```

“如果延迟超过几秒，”黛琳警告道，“系统就会认为服务有问题，直接杀掉。”

## 7.1.12.4 特殊情况

“有些特殊情况不会触发超时，”黛琳继续说道。

```kotlin
// 这些情况下不会超时：
// 1. 服务已经绑定到某个Activity
// 2. 服务有pending result要返回
// 3. 服务有前台Service类型的子服务在运行
```

“比如你的音乐服务正在播放，用户正在看播放界面，”黛琳举例道，“这种'活跃'的状态下，系统不会超时。”

## 7.1.12.5 测试超时

“那怎么测试超时呢？”希尔问。

“可以用开发者选项，”黛琳说道。

```kotlin
// 模拟超时：使用adb命令

// 或者用代码模拟

    // 观察系统是否杀掉服务
}
```

## 7.1.12.6 专业技术总结

本章我们学习了前台服务超时机制。

**核心要点：**

1. **Android 12+有超时限制** - 一般是20秒到几分钟
2. **超时后调用onTimeout()** - 可以选择继续或停止
3. **尽快调用startForeground()** - 避免超时
4. **活跃状态不会超时** - 有绑定或正在交互

**超时处理流程：**

```
启动Service → 等待超时 → onTimeout回调 → 创建通知继续 OR 停止服务
```
