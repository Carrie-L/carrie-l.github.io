# 第三部分：Android 官方核心领域（教学内容覆盖）

5. **Intent 和 intent 过滤器** (Intents and intent filters) -- 显式/隐式 Intent、Broadcast、PendingIntent [https://developer.android.com/guide/components/intents-filters]
6. **后台任务** (Background work) -- WorkManager、Handler、Executor [https://developer.android.com/develop/background-work/background-tasks]
7. **服务** (Services) -- Foreground Service、Background Service、Bound Service [https://developer.android.com/develop/background-work/services]
8. **闹钟** (Alarms) -- AlarmManager、精确闹钟、省电模式 [https://developer.android.com/develop/background-work/services/alarms]
9. **音频和视频** (Audio and video) -- MediaPlayer、ExoPlayer、CameraX [https://developer.android.com/media/audio-and-video]
10. **摄像头** (Camera) -- Camera2 API、CameraX、图像捕获 [https://developer.android.com/media/camera/choose-camera-library]
11. **传感器** (Sensors) -- SensorManager、加速度计、陀螺仪 [https://developer.android.com/develop/sensors-and-location/sensors/sensors_overview]
12. **用户位置** (User location) -- FusedLocationProvider、地图集成 [https://developer.android.com/develop/sensors-and-location/location]
13. **连接** (Connectivity) -- Retrofit、OkHttp、网络状态监听 [https://developer.android.com/develop/connectivity/overview]
14. **应用兼容性** (App compatibility) -- 适配不同 Android 版本、兼容性库 [https://developer.android.com/guide/app-compatibility]
15. **App Bundle** (App Bundle) -- AAB 原理、动态分发、Play Feature Delivery [https://developer.android.com/guide/app-bundle]

## B. 工具与工作流（6 个）

16. **编写与调试代码** (Write and debug code) -- Android Studio 调试器、断点使用、Logcat、内存分析 [https://developer.android.com/studio/write]
17. **构建项目** (Build projects) -- Gradle 构建系统、构建变体、模块化、增量构建 [https://developer.android.com/build/gradle-build-overview]
18. **测试应用** (Test your app) -- 单元测试、集成测试、UI 测试（Espresso）、测试金字塔 [https://developer.android.com/training/testing]
19. **性能优化** (Performance) -- 性能分析器、内存泄漏检测、UI 渲染优化、功耗分析 [https://developer.android.com/topic/performance/overview]
20. **命令行工具** (Command-line tools) -- adb、sdkmanager、avdmanager、构建命令行 [https://developer.android.com/tools]
21. **Gradle 插件 API** (Gradle plugin API) -- 自定义插件、任务配置、构建变体策略 [https://developer.android.com/reference/tools/gradle-api]

# 第十部分：叙事风格精炼指南（v1.1 补充，基于 1.1.1 章节修订经验）

> 本部分是对第七部分写作风格的具体补充，来自实际章节修订中总结的经验。写作时必须同时遵守。

## 氛围感：像夏日微风，不像教科书

* **场景要有温度和细节**：不只是"篝火旁"，而是"蝉鸣声一浪一浪地从远处树梢涌来"、"头顶被晚霞染成蜜桃色的天空"、"草地上的露水开始悄悄凝结"。用感官细节（声音、气味、触感、光线）让场景活起来。
* **季节感要鲜明**：每个大章节应有明确的季节基调（夏日蝉鸣、秋日落叶、冬日雪景、春日新绿等等AI自由发挥），并在场景描写中自然体现，不要只写"夜晚"或"白天"。
* **避免静态开场**：不要以"四个女孩坐在篝火旁"这种静态描述开场，要从一个正在发生的动作、声音或情绪切入。

## 对话：自然、俏皮、有个性

* **洛芙的台词要有少女感**：她的问题应该带着生活化的类比，例如"就像我记住你今天穿了一件好看的裙子那种记住？"而不是"数据存储是什么？"
* **每个角色有自己的说话方式**：
  * 黛琳：温柔笃定，像在分享一个理所当然的道理，不急不缓，极客少女，《摇曳露营》凛酱内核。
  * 伊莎：温柔诗意、比喻丰富，说话像在念一首小诗，猫娘角色。
  * 希尔：活泼直接，爱打响指，推动节奏，直率，朋克少女。
  * 洛芙：好奇、略呆，常用"哦哦哦！"、"哇……"、"就是那种感觉？"来回应，《摇曳露营》抚子内核。
* **小动作和神态要写进去**：角色的动作（揉头发、拍肩膀、把草茎绕在指间、舔冰棍等等自由发挥）让读者感觉真的在现场，而不是在读对话剧本。

## 技术讲解：比喻先行，代码收尾

* **每个概念的比喻要具体、可感知**：不要说"就像一个容器"，要说"就像你行李箱最底层压着的东西，不会轻易消失"。
* **比喻和代码之间要有过渡**：不要直接从比喻跳到代码块，中间要有一句"代码写起来也很简单--"或"希尔把电脑转过来让大家都能看见"这样的过渡。
* **代码讲解后要有角色反应**：代码块之后，洛芙或其他角色要有一句自然的反应（"哇，好像没有我想象的那么可怕嘛！"），不要让代码块孤立地悬在那里。
* **代码只是用来展示，但思路必须用角色语言说出来**：展示而非说教原则。

## 练习题：可执行 Task，不是题目列表

* 每道练习题必须是完整的 Task 卡（见第六部分格式），包含目标、步骤、验收标准、提示代码。
* Task 之间可以有依赖关系（如"先完成 Task 1 再做 Task 5"），形成自然的学习链条。
* 面试题改为开放式问答风格（"用自己的话说清楚"），更贴近真实面试场景。

## 日记本：有余韵，不只是总结

* 洛芙的日记不只是"今天学了 XXX"的流水账，要有一句带哲学感或情感余韵的话，例如"黛琳说得对，先想清楚用户需要什么，再选合适的方式--这好像不只是写代码的道理。"
* 日记语气要口语化、有洛芙的个人色彩，不要像总结报告。

# 第十一部分：叙事深度精炼指南（v1.2 补充，基于 1.2.1 章节修订经验）

> 本部分是对第十部分的进一步补充，来自 1.2.1 章节多轮修订中总结的经验。核心命题：**在技术教学与文学质感之间找到精确平衡点**，既不像技术博客，也不像注水小说。参考标杆：《摇曳露营》的安静呼吸感 × 《绿山墙的安妮》的情感丰富度 × 宫崎骏动画的自然观。

## 一、对话的情绪温度：不是念台词，是在生活

* **绝对不解释潜台词（最重要的一条）**：当一个动作或姿态已经传达了情绪，**不要在后面加一句解释它为什么传达了这个情绪**。让读者自己感受。

  ✅ "双手把温热的杯子捧在胸前。"（动作本身就说明了一切）
  ❌ "双手把温热的杯子捧在胸前，好像这样能捂暖刚才受伤的自尊心。"（多出的半句把留白杀死了）

  ✅ "黛琳永远是热白开水。"（读者自然会感受到这像她的为人）
  ❌ "黛琳永远是热白开水，这件事洛芙觉得非常像她本人，温热、透明、恰到好处。"（解释了比喻是什么意思）

  ✅ "洛芙愣了一下，然后慢慢地笑了。"（笑本身就是回应）
  ❌ "那是一种很安静的笑--不是答对题的得意，也不是被夸奖的害羞，而是一个人在清晨的山间营地里，忽然觉得自己好像真的往前走了一步的那种踏实。"（用一整段解释一个笑容）

* **情绪要有弧线**：一段教学对话中，角色的情绪应该有起伏--从困惑到恍然大悟，从挫败到振作，从紧张到放松。但情绪的呈现必须通过动作和对话，而不是旁白解说。
* **Montgomery 式情感表达**：用具体的身体感受表达抽象情绪，但**只写感受本身，不解释感受背后的情绪**。不说"她很开心"，说"阳光照在她脸上痒痒的"；不说"她难过"，说"晨风吹过来，把她的刘海拂到了眼睛上，她也没去拨"。注意这两个例子都没有在后面加"--这是因为她很开心/难过"。

## 二、嵌入式氛围感：一句足矣，不要整段

* **黄金法则**：氛围感融入动作和对话的间隙，每次只用**一句话**。在角色的动作中嵌入环境细节，而不是专门用一段来描写风景。

  ✅ "希尔把可可递过来，蒸汽在晨雾中慢慢散开"（氛围嵌入动作）
  ✅ "晨光从树缝间漏下来，在键盘上投下斑驳的光影"（一句带过）
  ✅ "一只蜻蜓掠过她的屏幕上方，翅膀在晨光里闪了一下"（代码块后的呼吸点）

  ❌ 连续两段以上的纯风景描写（节奏会断）
  ❌ 每一次对话都加一句环境描写（会变成套路）

* **时间线作为隐形节奏线**：用环境变化暗示时间流逝（薄雾→晨光→太阳升高→雾散），读者能感觉到"一个上午过去了"，不需要写"过了一会儿"。
* **感官要多样**：不要只写视觉，要轮换使用听觉（山雀叫）、嗅觉（咖啡焦香、松针清甜）、触觉（凉丝丝的空气、温热的杯子）、温度感（阳光晒暖肩膀）。

## 三、代码注释规范

* 代码中的注释采用「简洁总结 + 细节解释」结构：第一行注释用一句话概括这段代码在做什么，后续注释补充 API 参数、返回值、注意事项等细节。
* 注释对象是"看代码学习的读者"，假设他们第一次见到这个 API。
* 不在注释中使用比喻（比喻由小说正文中的角色对话承担），注释只负责精确的技术说明。
* 示例：
  ```kotlin
  // 用流写入文件（覆盖写）
  // openFileOutput() 是 Context 提供的方法
  // 参数1：文件名（不含路径，文件自动存入 filesDir 目录）
  // 参数2：文件模式。MODE_PRIVATE 表示私有，每次写入覆盖原内容
  ```

## 四、章节结尾：小说的余韵

* 技术总结 `---` 分隔线之前的最后几段是**小说正文的结尾**，必须有文学收束感，不能在代码块或 Logcat 输出之后突然结束。
* 结尾应包含：
  1. **一个安静的动作**：洛芙靠到椅背上、仰头看天、闭上眼睛--给读者一个"呼出一口气"的时刻。
  2. **一句点题的话**：由某个角色说出本章的核心感悟，但不是总结知识点，而是升华为一个更普世的感悟。例如："数据应该待在哪里，取决于它要活多久"--既是技术真理，也是生活隐喻。
  3. **一个回到自然的镜头**：结尾回到环境（山雀又叫了、太阳暖了、风吹过来了），和章节开头的环境形成呼应，像一个完整的圆。

## 五、角色一致性与世界观连贯性：跨章节守护

* **犯错的角色必须符合人设**：初学者（洛芙）犯初学者的错，老手（希尔、黛琳）不犯低级错误。如果需要展示一个知识点，让洛芙去踩坑，让老手来纠正。
* **角色的知识水平要连续**：如果角色在 1.1.1 里教过 Room，那 1.2.1 里她不可能不知道怎么保存文件。角色的成长是单向的，不能"忘记"之前已经展示过的能力。

### plot_summary 连贯性规则（强制）

**写作前必须读取"上一章"的 `plot_summary` YAML，以此为基础设定本章的时间/地点/季节/环境。**

#### "上一章"的确定规则

"上一章"指的是**同一卷中、章节号紧接在本章之前的那一章**，而非时间上最近写完的那一章。

- **同卷内**：6.1.2 的上一章是 6.1.1，13.1.12 的上一章是 13.1.11。
- **卷首章节**（如 6.1.1）：上一章是**上一卷（第5卷）的最后一章**。
- **上一卷未写时**：如果上一卷的最后一章尚未完成（写作顺序不一定按卷序），则本章作为**本卷开篇**，可以自由定义本卷的时间、地点、季节和环境--但必须在全书的整体时间轴上保持合理（不与已写章节矛盾）。
- **并行写作**：即使多章并行写作（如同时写 6.1.1、11.2.1、6.1.3），6.1.2 始终看 6.1.1 的 plot_summary，不看 11.2.1 或 6.1.3 的。

#### 连贯性硬性规则

1. **季节只能前进，不能后退**：秋→冬→春→夏 是自然推进。绝不允许秋→冬→秋这样来回跳跃。如果想延续同一季节，保持一致即可。
2. **地点转换需要正文交代**：如果上一章在"山林露营地"，本章想换到"湖畔"，必须在正文开头用 1-2 段描写旅途或场景转换。不能无说明地瞬移。
3. **时间自然衔接**：上一章下午 → 本章可以是傍晚/夜晚/次日清晨。不能从下午跳回到上午。
4. **环境延续**：上一章枫叶红了 → 本章不能突然大雪纷飞（除非正文中有 "一夜之间气温骤降" 等过渡）。

**输出时**：本章 YAML 必须包含 `volume_priority`、`volume_grade`、`chapter_importance`（见上表），以及 `plot_summary`：
```yaml
plot_summary:
  time: '...'         # 必填：时间段（如"秋日傍晚"）
  location: '...'     # 必填：具体地点
  scene: '...'        # 必填：场景描述
  season: '...'       # 必填：季节（必须与上一章衔接）
  environment: '...'  # 必填：环境氛围描写
  topic: '...'
  discussion: '...'
  problem_solved: '...'
  difficulty: '...'
  next_topic: '...'
```

### 分级与重要程度字段（必填，用于学习路线）

根据「在职 Android 开发 → 高级工程师」学习路线，每章 YAML 必须包含：

| 字段 | 含义 | 取值 |
|------|------|------|
| `volume_priority` | 卷的推荐学习顺序 | 1-21（1=最先学，21=最后学） |
| `volume_grade` | 卷级别 | `'A'` 必学 / `'B'` 可选 |
| `chapter_importance` | 章节在卷内的重要程度 | 1-5（5=核心必读，1=可选查阅） |

**卷级映射**（来自《完整小说目录》）：卷16→1, 卷17→2, 卷18→3, 卷2→4, 卷5→5, 卷6→6, 卷7→7, 卷1→8, 卷19→9, 卷4→10, 卷14→11, 卷15→12, 卷8→13, 卷13→14, 卷12→15, 卷3→16, 卷10→17, 卷9→18, 卷11→19, 卷20→20, 卷21→21。

**卷内 A/B 与 chapter_importance**：卷内 A 级章节填 5，B 级填 3，过渡章节填 4。具体规则见《完整小说目录》「卷内 A/B 建议」。

### status 字段（必填，不可省略）

**status** 是 YAML frontmatter 的必填字段，用于标记章节生命周期：

| 状态 | 含义 | 何时设置 |
|------|------|----------|
| `draft` | 骨架/未写 | 新建章节默认 |
| `done` | AI 已写完 | **AI 写完章节后必须立即改为 `done`** |
| `published` | 已审查通过并 push | 审查通过且 push 后由 Main Agent 改为 `published`，此后不再参与 review |

**规则**：AI 写完后都要把 status 改成 `done`；status 字段必须要有；审查通过且 push 的章节，status 改为 `published`，不再参与后续 review。可新增字段，不可删掉已有字段。
