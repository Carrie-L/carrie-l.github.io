## 核心写作原则：三分风景，七分技术（🔥 地狱严格版）
1. **纯粹肯定句原则（Syntax Hygiene）**：
   - **严禁**使用“不是……而是……”（Nothing is..., but is...）这种辩证法式的说教句式。
   - **PASS**：“DataStore 是一个全新的设计。”
   - **FAIL**：“DataStore 不是修修补补，而是全新的设计。”
2. **技术名词绝对纯粹化（降低心智负担）**：
   - 严禁对专业术语（如 Index, unique, ForeignKey, @Relation 等）进行文学化的过度比喻。术语必须直接、专业地呈现。代码逻辑必须像几何题一样锐利。
   - 术语必须直接、准确地呈现。
   - 代码逻辑解释必须像数学题一样锐利、清晰。
   - **FAIL 样例**："ForeignKey 就像是连接两个灵魂的红线..."（太玄学）
   - **PASS 样例**："ForeignKey 是表与表之间的锚点，保证了数据的一致性。"（准确）
3. **场景化的精准机制映射**：
   - 允许使用生动且极其贴切的生活场景来解释技术逻辑（例如：“一对一是加了唯一锁的门，一对多是取掉了锁的同一扇门”）。**如果你不能保证比喻是神来之笔，就干脆别用。**
4. **诗意留给环境与情绪（创造呼吸空间）**：
   - 将蒙哥马利式的白描（如跳跃的光影、微风、植物的气息、少女的微表情和肢体动作）作为知识点之间的转场和情绪铺垫。
   - 在高密度的技术讲解后，插入一两句环境描写，让读者的大脑得以短暂放松。
   - **推荐样例**："清晨的雾气还没有完全散去，屏幕在薄雾中泛着幽幽的蓝光" / "洛芙盯着屏幕，窗外的白桦树叶在风中沙沙作响"。
5. **角色与对话设定**：
   - 角色讨论代码时要专注、一针见血。遇到 Bug 会有真实的抓狂，跑通代码时会有明媚的喜悦。对话要干脆利落，绝不拖沓注水。
   - **禁止**生造增加理解成本的玄幻比喻。
   - **正文描写 vs 对话描写**：
    - **正文（旁白）**：多用环境描写烘托氛围（"午后的风把帐篷门帘吹得轻轻摆动"）。
    - **对话（角色）**：少用辞藻华丽的比喻，多用大白话和动作交互。
    - **FAIL（对话）**："希尔说：'代码就像这风中的落叶……'"（矫情）
    - **PASS（对话）**："希尔指着屏幕：'这行代码是多余的，删掉。'"（干脆）
6. 💬 对话自然生动，像是真的有几个少女在讨论一样

### 🔗 章节连贯性（🚨 硬性 FAIL — 与上一章 plot_summary 对齐）

**这是红线，违反直接 FAIL。**

审查时必须读取**上一章**的 YAML `plot_summary`，然后检查本章是否与之衔接。

**"上一章"定义**：同一卷中章节号紧邻本章之前的章节（如 6.1.2 的上一章是 6.1.1）。卷首章节（如 6.1.1）应看上一卷最后一章。如果上一卷最后一章尚未完成，则本章视为"本卷开篇"，允许自由定义时间/地点/季节（此时跳过季节连贯检查，但本卷后续章节仍需与此衔接）。

检查项：

1. **季节连贯**：上一章秋季 → 本章不能突然变冬季再下一章又回秋季。季节可以推进（秋→冬），但不可来回跳。
2. **地点连贯**：上一章在富士山麓露营场 → 本章不能无说明地出现在完全不同的地方。场景转换必须在正文中有过渡描写。
3. **时间连贯**：上一章是下午 → 本章可以是傍晚/夜晚/次日清晨，但不能无缘无故跳回上午。
4. **环境氛围连贯**：天气和氛围应自然延续或过渡，不能前一章晴空万里后一章暴风雪而不做任何交代。

**检查方法**：
- 读取本章 YAML 中的 `plot_summary`
- 与上一章的 `plot_summary` 逐字段对比
- 如有变化（如季节推进、地点转移），正文中必须有自然的过渡描写
- **🔴 FAIL**：上一章秋季山林 → 本章冬季小木屋 → 下一章又秋季山林（季节来回跳跃）
- **🔴 FAIL**：`plot_summary` 缺失 `time`/`location`/`season` 任一字段
- **🟢 PASS**：上一章秋季午后 → 本章秋季傍晚（自然推进），或秋季→初冬（正文中有"山间的风已经带着冬天的味道了"这样的过渡）

- 是否包含 plot_summary YAML frontmatter (time, location, season, environment 必填)？
- 是否包含 volume_priority、volume_grade、chapter_importance（分级与重要程度，见《完整小说目录》）？
- **status 字段（必填）**：AI 写完后必须设为 `done`；审查通过且 push 后改为 `published`，不再参与后续 review。status 可新增不可删。
- 是否包含 技术总结, 动手练习 (Task), 面试热身?
- **章节末尾结构顺序**（必须按此顺序）：技术总结（核心机制定义 → 结构图 → 反模式 → 设计哲学 → 动手练习 → 面试热身 → 参考实现要点）→ 学习建议 → 洛芙的小小日记本 → 今日关键词
- **章节质量自检（🚨 硬性 FAIL）**：自检全部达成后必须删除该节。**最终章节不应包含"章节质量自检"**，保持正文干净。若存在则 FAIL。
- **分割线规则（🚨 硬性 FAIL）**：`---` 分割线**仅允许三处**：① 专业技术总结前（小说正文与技术总结之间）、② 动手练习前（总结与实践之间）、③ 学习建议前（实践与结尾之间）。**小说正文内、Task 之间、其它位置一律禁止**。正文场景切换用空行或自然过渡，不得使用 `---`。
- **今日关键词**必须放在日记本之后（章节最末尾），列出本章正文中出现的**所有**专业术语，不只是核心概念。
- **动手练习**两种组织方式均可：A) 项目制（一个小项目拆分 8-10 个递进 Task）；B) 独立练习制（8+ 个独立 micro task）。核心原则：具体、可操作、可执行、描述详细到跟着操作就能有结果。允许与之前章节练习有重复——重复是学习的最好方式。
- 每个 Task 必须包含四部分：目标 + 步骤 + 验收标准 + 提示代码。
- 是否包含 面试热身 (Q1-Q5)？
- Task 是否足够明确且可执行？能直接按照指示跟练操作？
【非常重要、必须严格遵守的首要原则，内容必须和 章节md文档开头的yaml里的`official_url`的内容和结构保持对齐，将其改编为技术小说，必须按照官方文档上的内容来写！是根据官方文档改编！！！不能有遗漏的知识点！！！ 】

### 🎧 隐形倾听者原则（沉浸感测试）
整体是否看上去就像几个少女真的在野外露营地一边露营享受美景和生活，一边讨论代码和困惑一样？是否能给读者沉浸感，将读者带入那片风景优美的露营地，自然地成为其中一员的“隐形倾听者”？
- **去代码测试**：假如把所有代码块删掉，剩下的文字能否构成一段流畅、自然的对话？是否存在"为了引出代码而强行说话"的尬聊？
- **环境融合**：角色是否真的在露营？还是只是在一个名为"露营地"的会议室里开会？
    - **FAIL**：希尔说："下面我演示一下代码。"（会议室发言）
    - **PASS**：希尔把沾着饼干屑的手指在键盘上敲得噼啪作响，"看，把这行删掉试试。"（生活化动作）
- **读者代入**：文字是否能让读者感觉自己是坐在旁边石头上的第五个人？

### 🌅 结尾：通感与余韵 (Sensory Resonance)
- **拒绝课堂总结**：不要让角色口头念叨"今天我们学了什么"，而要让她们**做**点什么来体现学到的东西。
- **身心通感**：将抽象的代码哲理与具体的身体/感官体验联系起来。
    - **FAIL**："今天学会了异步查询，我很开心。"（干瘪）
    - **PASS**："洛芙咬了一口热乎乎的烤大福，那种'轻飘飘'的口感，就像把所有重活都丢给了后台线程一样轻松。"（通感：口感 <-> 代码特性）
- **景语收束**：最后一段留给环境，让技术思考消融在某种自然氛围里（日落、篝火、星空）。
