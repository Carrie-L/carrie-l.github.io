协程的结构化并发（Structured Concurrency）通过 `CoroutineScope` 建立父子层级，保证子协程的生命周期不超过父协程。默认情况下，任意一个子协程抛出未捕获异常会向上传播，取消整个作用域内的所有兄弟协程——这在"全部成功或全部回滚"的事务场景非常有用。

但实际业务中，更常见的需求是**独立并发**：某个请求失败不应波及其他正在执行的任务。此时需要 `supervisorScope` 或在构建 scope 时传入 `SupervisorJob()`。

```kotlin
// 普通 coroutineScope：子协程互相牵连
coroutineScope {
    launch { uploadLog() }      // 若此处抛异常
    launch { fetchRecommend() } // 此协程也会被取消
}

// supervisorScope：失败彼此隔离
supervisorScope {
    launch {
        runCatching { uploadLog() }
            .onFailure { log(it) }
    }
    launch { fetchRecommend() } // 独立运行，不受影响
}
```

在 ViewModel 中，`viewModelScope` 底层使用 `SupervisorJob + Dispatchers.Main`，因此每个 `launch {}` 块相互独立、互不干扰。但需注意 `async` 的异常陷阱——若 `Deferred` 从未被 `await()`，异常会被静默吞掉，形成隐患；应始终在 `async` 块内用 `runCatching` 捕获，或在 `await()` 处处理。

**经验法则**：业务层用 `supervisorScope` 保持弹性，基础设施层（数据库事务、批量写入等需原子性的场景）用普通 `coroutineScope` 保持一致性。两者配合，才是真正健壮的并发设计。
