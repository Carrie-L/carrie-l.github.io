将冷流转为热流是 Android 架构中的常见需求，`stateIn` 与 `shareIn` 是两个核心操作符，理解其差异对于构建健壮的数据层至关重要。

`stateIn` 将流转换为 `StateFlow`，始终持有最新值，新订阅者立即收到当前状态，天然适合 UI 状态管理。配合 `WhileSubscribed(5000)` 使用效果最佳：

```kotlin
val uiState: StateFlow<UiState> = repository.dataFlow
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5000),
        initialValue = UiState.Loading
    )
```

`shareIn` 将流转换为 `SharedFlow`，无内置状态概念，适合一次性事件流（导航跳转、Toast 通知）。设置 `replay = 0` 确保事件不被重放给新订阅者：

```kotlin
val navEvent: SharedFlow<NavEvent> = _navEvents
    .shareIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(),
        replay = 0
    )
```

**关键技巧**：`WhileSubscribed(5000)` 在最后一个订阅者离开后延迟 5 秒才停止上游。这个 5 秒窗口正好覆盖屏幕旋转的配置变更时间，避免重复发起网络请求，是高级 Android 工程师必掌握的性能细节。

**选择原则**：需要"当前值"概念、支持 `.value` 访问 → 用 `stateIn`；纯事件流、不需回放、只通知活跃订阅者 → 用 `shareIn`。
