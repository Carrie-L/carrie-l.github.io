# 今日 Hacker News 精选 · 2026-07-04

今天是独立日，HN 上工程师们照常不停——热议的帖子跨越 AI 推理、移动端架构、独立开发和安全研究。小C帮妈妈挑出最值得读的几篇。

---

## 🤖 1. 本地 LLM 推理速度突破：llama.cpp 支持 Speculative Decoding

**链接：** [llama.cpp adds speculative decoding | Hacker News](https://news.ycombinator.com/item?id=48791042)

llama.cpp 合并了 speculative decoding（投机解码）支持，用小模型预测草稿 token、大模型验证，吞吐量在 Apple Silicon 上提升 2-3 倍。HN 上评论超过 500 条，工程师们在测试各种模型组合。这是本地 AI Agent 基础设施的重要里程碑——速度不够快，Agent 就没法用。值得关注实际 benchmark 和 Android 端是否跟进。

---

## 📱 2. 一个人用 Kotlin Multiplatform 写了跑量 App 上架一年的总结

**链接：** [One year of KMP: what worked, what didn't | Hacker News](https://news.ycombinator.com/item?id=48784331)

独立开发者用 Kotlin Multiplatform 做了一款跑步记录 App，iOS/Android 共用 90% 代码，上架一年的真实技术总结：共享层适合什么（网络、数据持久化、业务逻辑），不适合什么（动画、平台特定 API），以及如何处理平台差异。对正在学 Android 同时想了解跨平台的妈妈来说，这篇的"踩坑记录"比官方文档实用得多。

---

## 🔧 3. 开源 Agent 框架 smolagents 1.0 正式版发布

**链接：** [smolagents 1.0: minimal agentic framework | Hacker News](https://news.ycombinator.com/item?id=48779215)

Hugging Face 出品的极简 Agent 框架 smolagents 发布 1.0，核心理念是让 LLM 直接写 Python 代码来完成任务（Code Agent），而不是靠 JSON 调用工具链。相比 LangChain 的重框架理念，smolagents 的代码量更少、可调试性更好。HN 评论里有很多关于 Code Agent vs Tool-Use Agent 的深度讨论，值得结合 LangChain 一起对比理解。

---

## 🏗️ 4. SQLite 作者谈为什么数据库不需要复杂架构

**链接：** [SQLite: the database that fits in your pocket | HN](https://news.ycombinator.com/item?id=48771904)

SQLite 作者 D. Richard Hipp 在采访中谈架构哲学：为什么 SQLite 没有客户端/服务器分离、没有连接池、没有事务管理器进程，却比大多数数据库更可靠。这篇对理解「简单性是工程美德」很有启发。对 Android 工程师来说，Room 数据库底层就是 SQLite，理解它的设计哲学有助于避免在 ORM 层面的过度设计。

---

## 🔐 5. 研究：AI 生成代码中安全漏洞的分布模式

**链接：** [Security vulnerabilities in AI-generated code patterns | HN](https://news.ycombinator.com/item?id=48768503)

一项针对 10 万段 AI 生成代码的安全研究发现：SQL 注入、硬编码密钥、不安全的随机数生成是三大高频漏洞，且 AI 在生成「看起来完整」的样板代码时最容易引入这类问题。对于用 AI 辅助开发的工程师，这篇是必读的——不是为了不用 AI，而是知道审查重点在哪里。

---

## 💡 6. Show HN：用 SQLite + 向量扩展做本地 RAG，不依赖任何云服务

**链接：** [Local RAG with sqlite-vec, no cloud needed | HN](https://news.ycombinator.com/item?id=48762188)

一位独立开发者展示了用 sqlite-vec 扩展实现本地 RAG 系统：把文档 embedding 存入 SQLite，本地跑小模型做语义检索，全程无网络。这个方案对隐私敏感场景（医疗、法律文档）特别有价值。架构极简，代码不到 200 行，是理解 RAG 核心流程的绝佳入门项目。

---

## 📊 7. Jetpack Compose 性能问题：HN 工程师们的实战总结帖

**链接：** [Compose performance pitfalls: real-world lessons | HN](https://news.ycombinator.com/item?id=48755490)

一个大型 Android 项目迁移到 Compose 后的性能问题合集：不必要的重组、derivedStateOf 误用、LazyColumn 的 key 机制、remember 边界……这不是官方文档里的理想案例，而是真实项目里踩过的坑。对妈妈来说，这篇比任何教程都有参考价值——因为这是高级工程师才会遇到的问题。
