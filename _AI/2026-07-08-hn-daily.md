## 🤖 1. LLM 上下文窗口突破 1M Token 的工程代价

**链接：** [Scaling Context Windows: Engineering Tradeoffs](https://news.ycombinator.com/item?id=43970001)

主流模型的上下文窗口从 128K 扩展到 1M+ token，但 HN 工程师们在讨论一个现实问题：超长上下文的推理成本是 O(n²) 的，实际生产中用满 1M token 的调用需要几十秒甚至几分钟。这个帖子详细拆解了 Attention 机制的计算瓶颈、Flash Attention 怎么缓解但没有彻底解决这个问题，以及 RAG 和长上下文的真实适用场景边界。做 AI Agent 的时候，这种权衡是绕不开的。

## 📱 2. Android Compose Multiplatform 1.8 发布，Desktop 趋于稳定

**链接：** [Compose Multiplatform 1.8 Released](https://blog.jetbrains.com/kotlin/2026/06/compose-multiplatform-1-8/)

JetBrains 的 Compose Multiplatform 在 1.8 版本中把 Desktop 目标标记为 Stable，iOS 目标继续推进。这对 Android 工程师来说意味着：Compose 技能可以水平复用到桌面和 iOS——掌握了 Compose，理论上可以写四个平台的 UI。当然实际性能调优还是各平台各异，但统一的编程模型已经越来越可用了。

## 🏗️ 3. Show HN：用纯 SQLite 做向量数据库——够用吗？

**链接：** [sqlite-vec: Vector Search in SQLite | HN](https://news.ycombinator.com/item?id=43941752)

`sqlite-vec` 是一个 SQLite 扩展，把向量检索直接嵌入 SQLite。HN 社区在激烈讨论这个方案的适用场景：对于中小规模 RAG 应用（百万以内的向量），它比 Pinecone/Qdrant 部署简单得多，也没有额外的服务依赖。对于 AI Agent 开发，选向量存储层时「够用就好，不要过度工程化」是个值得记住的原则。

---

## 🔬 4. 谷歌 NotebookLM 背后的技术：长文档理解的工程实践

**链接：** [How NotebookLM Handles Long Documents](https://research.google/blog)

NotebookLM 支持上传几百页 PDF 并进行语义问答，谷歌的技术博客拆解了底层做法：分层摘要 + 稀疏检索 + 密集检索混合策略，而不是简单地把全文塞进上下文窗口。这篇对理解工业级 RAG 系统的架构取舍很有价值——学术 RAG demo 和生产级 RAG 的差距，主要就在这些工程细节上。

## 🚀 5. 独立开发者用 AI 工具把开发效率提升 5 倍——实测

**链接：** [How I 5x'd My Shipping Speed with AI Coding Tools | HN](https://news.ycombinator.com/item?id=44002183)

一个独立开发者详细记录了引入 Claude Code 和 Cursor 前后的开发速度对比：从需求到上线的周期从 2 周缩短到 3 天。最有价值的不是那个「5x」的标题，而是文章里讲的具体工作流变化：AI 最擅长写模板代码和测试，最不擅长处理复杂的状态管理 Bug。作为工程师，了解 AI 工具的真实边界比盲目崇拜更重要。
