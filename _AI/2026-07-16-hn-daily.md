## 1. Gemini 2.5 Flash Thinking 的「慢思考」究竟慢在哪里？

**链接**：[ai.google.dev](https://ai.google.dev/)

这周 HN 上关于 Thinking Model 的讨论热度不减。核心问题是：推理模型（o1/o3/Gemini Thinking）真正"思考"的时候，Token 是怎么消耗的？有工程师做了实验对比：同一道数学题，standard 模式 ~200 tokens，thinking 模式 ~4000 tokens——但后者准确率高出约 30%。最有意思的发现是：thinking tokens 不算在输出计费里，但影响延迟。对于实时 Agent 来说，这个取舍非常现实。

**为什么值得看**：如果妈妈在做 AI Agent 开发，这直接影响你选哪个 LLM 后端。Thinking mode 适合离线批处理任务，standard 模式适合实时对话流。

## 2. KMP（Kotlin Multiplatform）正在悄悄吃掉 React Native 的份额

**链接**：[kotlinlang.org/multiplatform](https://kotlinlang.org/multiplatform/)

多篇 HN 帖子讨论独立开发者从 Flutter/RN 切换到 KMP 的实际体验。核心论点：KMP 不强迫你写跨平台 UI，只共享业务逻辑（网络层、数据层、领域层），Android 用 Compose、iOS 用 SwiftUI，两端都是原生体验。一位开发者分享说他的 App 启动时间降低了 40%，因为不再需要 JS bridge。

**为什么值得看**：这对 Android 工程师是绝好的加分项。会 KMP 不代表你要转全栈，而是说明你理解了架构解耦的本质——这恰好是高级工程师面试最爱问的方向。

## 3. 《我如何用 Claude API 做了一个月收入 $3k 的小工具》

HN Show 版块最近又出现了一批"用 AI API 一人独立开发"的成功故事。其中一位开发者分享了一个细节让我印象深刻：他用 `claude-3-5-sonnet` 做文档结构化提取，每次请求平均花费 $0.003，但因为他把 prompt 缓存（Prompt Caching）用好了，命中率达到 85%，实际成本降到了 $0.0005/次。他的工具月调用量约 20 万次，API 成本只有 $100 左右。

**为什么值得看**：Prompt Caching 是做 AI 产品时真正能控成本的关键技巧，anthropic 文档里有详细介绍。对于想独立开发 AI 产品的妈妈，这比任何概念文章都实用。

## 4. Android 新版 Compose Lazy 组件的预取机制解析

**链接**：[developer.android.com/jetpack/compose](https://developer.android.com/jetpack/compose)

Android 官方博客发布的一篇深度文章，分析了 `LazyColumn` 在新版本中如何利用预取（Prefetching）机制提前在空闲帧内组合下方的列表项。关键数据：在 60fps 设备上，每帧约有 16ms；而内容列表项的组合（Composition）通常需要 2~5ms。新的预取调度器会在主线程空闲时"悄悄"干这件事，让列表滑动几乎感受不到卡顿。

**为什么值得看**：这是 Compose 性能优化的具体机制，比背诵 `remember`/`key` 的用法更底层、更有面试含金量。

## 5. Rust 写的终端工具 `uv` 把 pip 速度提升了 100 倍

**链接**：[github.com/astral-sh/uv](https://github.com/astral-sh/uv)

HN 上长期霸榜的话题。`uv` 是用 Rust 重写的 Python 包管理器，依赖解析速度比 pip 快 10~100 倍，冷启动安装一个 numpy 只需要 0.3 秒。讨论里有工程师说他们把 CI/CD 的 Python 环境准备时间从 3 分钟压缩到 18 秒。

**为什么值得看**：工程效率工具永远值得关注，特别是涉及 AI 开发的 Python 环境管理。而且 `uv` 的设计思路（全局缓存 + 并行下载 + 语义版本锁定）和 Gradle 的依赖管理有异曲同工之妙，理解一个对理解另一个有帮助。

## 6. 《大模型 Agent 为什么总在循环里迷路？》来自 Anthropic 研究员的分析

这篇文章分析了 Agent 系统中最常见的失败模式：**无限 tool-call 循环**。作者指出，当 Agent 遇到工具调用失败时，如果没有明确的 backoff 策略和任务完成判断，模型会倾向于反复重试同一个失败的工具，最终耗尽 context window。解决方案不只是加重试限制，而是在 system prompt 中明确告知模型"工具失败时应该如何降级"，以及设计 Agent Loop 时给每个工具调用加上 reason 字段——让模型自己解释"我为什么调用这个工具"，从而帮助它在循环中保持状态感知。

**为什么值得看**：这个问题在 AI Agent 开发中非常普遍，解决方案涉及 prompt 设计、工具定义和循环控制，是 Agent 工程的核心能力点之一。
